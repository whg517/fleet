## 第一部分：信息架构

### 设计原则

- **一级导航不超过 6 项**，按"日常使用频率"和"操作动线"排序
- 高频操作（看服务、部署、审批）放最前，低频管理操作放后面
- 不同角色登录后看到相同导航结构，但按权限渲染可见菜单项和操作按钮

### 一级导航结构

```
┌─────────────────────────────────────────────────────────────────────┐
│  📋 服务目录  │  🚀 部署中心  │  ✅ 发布审批  │  🔍 集群运维  │  📊 审计分析  │  ⚙️ 系统设置  │
└─────────────────────────────────────────────────────────────────────┘
```

### 各模块详细说明

#### 1. 📋 服务目录（Services）

> **主要使用者：** 全部角色（运维 / 开发 / PM）

| 二级页面 / Tab | 功能说明 | 权限 |
|---------------|---------|------|
| 全部服务 | 100+ 服务的列表，支持搜索、按团队/标签/部署类型过滤、按环境状态筛选。卡片视图 + 列表视图切换 | 全部可查看 |
| 我的团队 | 按当前用户所属团队过滤的服务视图，快速定位自己负责的服务 | 全部可查看 |
| 版本看板 | 跨环境版本矩阵（行=服务，列=dev/test/pre/prod），高亮版本不一致项。支持按团队分组、按版本差距排序 | 全部可查看 |
| 新服务接入 | 向导式服务接入流程（基本信息 → 关联 Chart/Role → 关联 Harbor → 初始化环境 → 完成） | operator / admin |
| 服务详情 | 点击服务进入详情页（见第三部分） | 全部可查看 |

#### 2. 🚀 部署中心（Deployments）

> **主要使用者：** 运维（全环境）、开发（dev/test 自助）

| 二级页面 / Tab | 功能说明 | 权限 |
|---------------|---------|------|
| 部署列表 | 全部部署记录列表，支持按服务/环境/状态/时间过滤。展示：服务名、版本、环境、状态、发起人、时间 | 全部可查看 |
| 新建部署 | 部署表单：选服务 → 选版本 → 选环境 → 填发布说明 → 提交（或审批）。支持批量部署 | dev/test 自助；pre/prod 需审批 |
| 构建管理 | 构建任务列表（Argo Workflows），展示构建状态和流水线进度。支持手动触发构建 | 全部可查看；开发可触发 |
| 镜像仓库 | Harbor 镜像列表，展示镜像版本、扫描结果（Trivy）。支持按漏洞等级过滤 | 全部可查看 |

#### 3. ✅ 发布审批（Approvals）

> **主要使用者：** 运维主管（pre/prod 审批）、PM（审批进度追踪）

| 二级页面 / Tab | 功能说明 | 权限 |
|---------------|---------|------|
| 待我审批 | 当前用户待处理的审批任务列表。每项展示：服务、版本、环境、提交人、提交时间、风险评估摘要 | 配置的审批人 |
| 全部审批 | 全部审批记录，支持按状态（待审批/已通过/已拒绝/已超时）过滤 | operator+ |
| 发布窗口 | 发布窗口和冻结期配置。展示当前窗口状态（开放中/冻结中）、规则列表 | admin 配置；全部可查看 |

#### 4. 🔍 集群运维（Cluster）

> **主要使用者：** 运维工程师

| 二级页面 / Tab | 功能说明 | 权限 |
|---------------|---------|------|
| 集群概览 | 集群健康总览：节点数、资源水位（CPU/内存总量/已分配/实际使用）、异常节点高亮 | 全部可查看 |
| 节点管理 | 节点列表，展示状态/资源使用率/Pod 数量。支持 cordon/drain/uncordon 操作 | 查看：全部；操作：operator+ |
| Pod 总览 | Pod 列表，支持按命名空间/服务过滤。展示状态、重启次数、节点、日志入口 | 全部可查看 |
| 配额管理 | ResourceQuota / LimitRange 管理页面，按环境配置资源配额上限和告警阈值 | 查看：全部；操作：admin |

#### 5. 📊 审计分析（Audit & Analytics）

> **主要使用者：** 运维（审计日志）、PM（分析报表）、安全审计员

| 二级页面 / Tab | 功能说明 | 权限 |
|---------------|---------|------|
| 操作日志 | 审计日志查询页，支持按人员/时间/服务/环境/操作类型筛选。展示操作详情、来源 IP、hash chain 校验状态 | admin / auditor |
| 发布统计 | 发布次数、成功率、平均耗时，按团队/服务/环境/时间维度统计，支持趋势图和导出 | operator+ |
| 回滚分析 | 回滚次数、回滚率统计，区分自动/手动回滚，回滚原因分类，关联告警事件 | operator+ |
| 权限矩阵 | 用户×服务×环境权限分配矩阵视图，支持僵尸权限检测、高权限人员筛选 | admin |
| 通知记录 | Webhook 通知发送记录，展示发送状态（成功/失败/重试）、通知内容、响应码 | admin |

#### 6. ⚙️ 系统设置（Settings）

> **主要使用者：** 运维主管 / admin

| 二级页面 / Tab | 功能说明 | 权限 |
|---------------|---------|------|
| 平台健康 | 依赖组件健康状态面板（PostgreSQL/Redis/Argo CD/Harbor/Git/Prometheus/OIDC），展示响应时间和状态 | 全部可查看 |
| 基础设施配置 | 系统级配置管理：OIDC 连接信息、Argo CD 地址、Harbor 地址、Git 仓库地址。支持连接测试 | admin |
| Webhook 配置 | Webhook 端点管理：创建/编辑/删除 Webhook，配置订阅事件、过滤规则、重试策略 | admin |
| 凭证管理 | 凭证列表（Git Token、Harbor 密码、kubeconfig），展示凭证状态和过期时间。支持轮换 | admin |
| 用户与角色 | 用户列表、角色管理、默认权限配置。支持禁用用户、调整角色 | admin |
| 通知模板 | 通知订阅规则配置（按服务/环境过滤、合并策略、优先级分级） | admin |

## 第四部分：K8s/Argo CD 术语翻译对照表

### 设计原则

- **所有面向用户的 UI 文案不出现原始 K8s/Argo CD 英文术语**
- 内部系统保留原始术语，仅 UI 层做翻译映射
- 翻译以"开发工程师能直接理解"为标准，不需要学习 K8s 概念

### 4.1 状态类（Status）

| K8s/Argo CD 原始术语 | 面向用户的中文翻译 | UI 图标 | 备注 |
|---------------------|------------------|---------|------|
| Healthy | 🟢 运行中 | 绿色圆点 | 服务正常运行 |

| Degraded | 🔴 异常 | 红色圆点 | 服务部分不可用 |

| OutOfSync | ⚠️ 配置偏差 | 橙色三角 | Git 配置与实际不一致 |
| Missing | ❓ 缺失 | 灰色问号 | 资源在 Git 中定义但集群中不存在 |
| Synced | ✅ 已同步 | 绿色对勾 | Git 与集群状态一致 |
| Unknown | ❔ 未知 | 灰色圆点 | 数据采集失败或超时 |
| Pending | ⏳ 等待中 | 沙漏图标 | Pod 等待调度或资源 |
| Running (Pod) | 🟢 运行中 | 绿色圆点 | Pod 正常运行 |

## 第五部分：错误提示三段式规范

### 5.1 标准格式

所有平台错误提示统一采用三段式结构：

```
┌─────────────────────────────────────────────┐
│  ❌ [错误标题]                                │
│                                              │
│  发生了什么：简明描述错误现象                  │
│                                              │
│  为什么：导致错误的可能原因                    │
│                                              │
│  下一步怎么做：可操作的修复步骤                │
└─────────────────────────────────────────────┘
```

**规则：**
- **发生了什么**：一句话说明当前状态，用户视角，不超过 30 字
- **为什么**：列出 1-3 个最可能的原因，每个不超过 20 字
- **下一步怎么做**：列出 1-3 个可操作步骤，包含按钮/链接引导
- 避免出现 K8s 原始错误码，使用第四部分的翻译表
- 如果错误可自动恢复（如重试），在提示中标注"系统将在 N 秒后自动重试"

## 第六部分：部署进度多阶段设计

### 6.1 六个阶段定义

| 阶段 | 名称 | 说明 | 数据来源 |
|------|------|------|---------|
| 1 | 提交 | 用户提交部署请求（pre/prod 进入审批中） | 平台创建 Deployment 记录 |
| 2 | 校验 | 前置校验：镜像存在性、Chart 可用性、权限校验、部署锁 | 平台 + Harbor + Git |
| 3 | 配置 | 更新 Git（values override）、更新 Argo CD Application | Git + Argo CD |
| 4 | 部署 | Argo CD sync 执行，Pod 滚动更新 | Argo CD API |
| 5 | 健康检查 | Argo CD 检查 Application 健康（readiness/liveness probe） | Argo CD API |
| 6 | 完成 | 部署成功，记录审计、发送通知 | 平台 |

### 6.3 阶段超时阈值

| 阶段 | 正常耗时 | 标黄警告 | 标红超时 | 超时动作 |
|------|---------|---------|---------|---------|
| 提交 | < 1s | — | > 10s | 标记失败 |
| 校验 | < 5s | — | > 30s | 标记失败 |
| 配置 | < 10s | — | > 60s | 标记失败，通知运维 |
| 部署（K8s） | 1-3min | > 3min | > 5min | 提示继续等待或回滚 |
| 部署（Ansible） | 2-5min | > 5min | > 10min | 提示检查主机 |
| 健康检查 | 30s-2min | > 3min | > 5min | 触发自动回滚（如配置） |
| 完成 | < 1s | — | — | — |

## 第八部分：操作确认规范

### 8.1 确认级别分类

| 级别 | 名称 | 确认方式 | 适用场景 |
|------|------|---------|---------|
| L0 | 无需确认 | 直接执行 | 查看类操作、搜索、筛选 |
| L1 | 普通确认 | 弹窗确认（点击"确认"按钮） | 可逆操作、dev/test 环境操作 |
| L2 | 强确认 | 输入服务名确认 | 不可逆操作、prod 环境高风险操作 |

### 8.2 普通确认（L1）

**格式：**

```
┌──────────────────────────────────────────┐
│  ⚠️ 确认操作                               │
│                                           │
│  即将执行：部署 user-service v1.2.3       │
│  到 dev 环境                              │
│                                           │
│  当前版本：v1.2.2                         │
│  目标版本：v1.2.3                         │
│                                           │
│  [取消]                    [确认部署]      │
└──────────────────────────────────────────┘
```

**适用操作列表：**

| 操作 | 确认文案 | 环境限制 |
|------|---------|---------|
| 部署到 dev | "即将部署 {service} {version} 到 dev 环境" | dev |
| 部署到 test | "即将部署 {service} {version} 到 test 环境" | test |
| 回滚（dev/test） | "即将回滚 {service} {env} 到上一版本 {version}" | dev/test |
| 配置变更（dev/test） | "即将修改 {service} {env} 的配置，涉及 N 项变更" | dev/test |
| 扩缩容（dev/test） | "即将将 {service} {env} 副本数从 X 调整到 Y" | dev/test |
| 手动触发构建 | "即将触发 {service} {branch} 分支的构建" | — |
| 冻结/解冻服务 | "即将{冻结/解冻} {service}（{env}环境）" | — |
| 批量部署 | "即将批量部署 N 个服务到 {env} 环境" | dev/test |
| 配置回退 | "即将回退 {service} {env} 配置到版本 v{N}" | dev/test |
| 重启实例（dev/test） | "即将滚动重启 {service} {env} 的所有实例" | dev/test |

### 8.3 强确认（L2）

**格式：**

```
┌──────────────────────────────────────────────────────────┐
│  🔴 高风险操作 — 需二次确认                                 │
│                                                           │
│  即将执行：{操作描述}                                      │
│  影响环境：🔴 prod                                         │
│                                                           │
│  ⚠️ 此操作不可逆 / 可能影响线上服务                        │
│                                                           │
│  请输入服务名 "{service-name}" 以确认：                    │
│  ┌────────────────────────────────────────┐               │
│  │                                        │               │
│  └────────────────────────────────────────┘               │
│                                                           │
│  [取消]                              [确认执行] 🔒         │
│                                                           │
│  💡 确认按钮在服务名输入正确后点亮                          │
└──────────────────────────────────────────────────────────┘
```

**规则：**
- 用户必须在输入框中手动键入完整服务名（大小写敏感）
- 输入正确前"确认执行"按钮为禁用状态（灰色）
- 服务名以平台注册的为准，展示在确认文案中供用户对照
- 确认操作记录审计日志，包含用户输入的服务名

**适用操作列表（强确认）：**

| 操作 | 确认文案 | 要求输入 | 理由 |
|------|---------|---------|------|
| 部署到 prod | "即将部署 {service} {version} 到 **prod** 环境，替换当前版本 {current_version}" | 输入服务名 | prod 部署影响线上用户 |
| 部署到 pre | "即将部署 {service} {version} 到 **pre** 环境" | 输入服务名 | pre 影响准生产验证 |
| 回滚 prod | "即将回滚 {service} **prod** 环境到版本 {target_version}" | 输入服务名 | prod 回滚影响线上 |
| 配置变更（pre/prod） | "即将修改 {service} **{env}** 的配置，涉及 N 项变更" | 输入服务名 | 配置变更可能导致服务异常 |
| 扩缩容 prod | "即将将 {service} **prod** 副本数从 X 调整到 Y" | 输入服务名 | 影响 prod 容量 |
| 节点 drain | "即将驱逐节点 {node-name} 上的 {N} 个实例到其他节点" | 输入节点名 | 影响该节点上所有服务 |
| 服务下线 | "即将下线 {service}，清理所有 K8s 资源和镜像，此操作**不可逆**" | 输入服务名 | 资源删除不可恢复 |
| 批量冻结 prod | "即将冻结 **prod** 环境的全部 {N} 个服务" | 输入 "prod-freeze" | 影响面大 |
| 删除 Webhook | "即将删除 Webhook 配置 {webhook-name}" | 输入 Webhook 名 | 可能导致后续通知丢失 |
| 更新系统配置 | "即将修改平台系统配置（{config-item}），可能影响平台行为" | 输入配置项名 | 影响平台自身 |
| 凭证轮换 | "即将更新 {credential-type} 凭证" | 输入凭证名 | 可能影响平台连接 |
| 强制释放部署锁 | "即将强制释放 {service} {env} 的部署锁（当前持有者：{user}）" | 输入服务名 | 可能导致并发冲突 |

### 8.4 prod 回滚特殊处理

prod 环境的**自动回滚**不需要用户确认（紧急止血优先），但**手动回滚到指定历史版本**需要强确认：

```
┌──────────────────────────────────────────────────────────┐
│  🔴 手动回滚到历史版本                                     │
│                                                           │
│  即将回滚 payment-service prod 环境：                      │
│                                                           │
│  当前版本：v3.2.0 (2026-07-11 22:00 部署)                 │
│  目标版本：v3.0.0 (2026-06-28 部署)                        │
│                                                           │
│  ⚠️ 跳过了 2 个中间版本（v3.1.0, v3.1.2）                 │
│  ⚠️ 目标版本与当前版本的数据库 schema 可能不兼容            │
│                                                           │
│  回滚原因（必填）：                                        │
│  ┌────────────────────────────────────────┐               │
│  │                                        │               │
│  └────────────────────────────────────────┘               │
│                                                           │
│  请输入服务名 "payment-service" 以确认：                    │
│  ┌────────────────────────────────────────┐               │
│  │                                        │               │
│  └────────────────────────────────────────┘               │
│                                                           │
│  [取消]                              [确认回滚] 🔒         │
└──────────────────────────────────────────────────────────┘
```

### 8.5 确认弹窗交互规范

1. **弹窗遮罩**：半透明遮罩层，点击遮罩不关闭弹窗（防止误触关闭）
2. **焦点管理**：弹窗打开后焦点自动移到输入框（强确认）或"确认"按钮（普通确认）
3. **键盘支持**：
   - `Enter` 键 = 点击确认按钮（仅普通确认）
   - `Esc` 键 = 点击取消按钮
4. **加载状态**：确认后按钮变为 loading 状态，防止重复提交
5. **超时关闭**：确认弹窗 5 分钟无操作自动关闭（取消本次操作）
6. **文案一致性**：所有确认弹窗的确认按钮使用"确认{操作名}"而非通用的"确定"

### 8.6 操作确认流程图

```
用户点击操作
     │
     ▼
  权限校验 ──失败──→ 显示权限不足提示（Error 7）
     │
   通过
     │
     ▼
  冻结校验 ──冻结──→ 显示服务冻结提示（Error 8）
     │
   未冻结
     │
     ▼
  前置校验 ──失败──→ 显示校验错误（Error 6）
     │
   通过
     │
     ▼
 操作级别判断
     │
     ├── L0（无需确认）──→ 直接执行
     │
     ├── L1（普通确认）──→ 弹窗确认 ──确认──→ 执行
     │                                  └──取消──→ 中止
     │
     └── L2（强确认）──→ 弹窗 + 输入服务名 ──输入正确 + 确认──→ 执行
                                                └──输入错误/取消──→ 中止
```
