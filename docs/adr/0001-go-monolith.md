# ADR-0001: 后端采用 Go 单体架构

## 状态

Accepted

## 背景

平台后端需要一个服务 100+ 微服务的管理平台。需要考虑：
- 并发编排能力（同时触发多个部署、轮询多个 Argo CD 状态）
- 部署运维复杂度（团队规模有限）
- 100+ 微服务的管理平台本身不需要微服务架构
- 需要快速迭代

## 决策

使用 Go 语言，单体架构起步。按 internal 域模块分包，单体内部保持清晰边界。

## 后果

### 正面
- 单二进制部署，运维简单
- Go 并发模型（goroutine）天然适合编排操作
- 编译型语言，类型安全
- 交叉编译方便

### 负面
- 单体规模增长后可能需要拆分（但当前规模不构成问题）
- Go 框架生态不如 Java 丰富

### 中性
- 团队需要 Go 经验

## 备选方案

| 方案 | 拒绝理由 |
|------|---------|
| Java | 部署重（JVM），运维成本高 |
| Python | 性能不如 Go，部署不如单二进制方便 |
| Go 微服务 | 过度设计，当前规模不需要 |
| Rust | 学习曲线高，开发效率低 |
