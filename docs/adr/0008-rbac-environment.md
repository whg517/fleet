# ADR-0008: 权限模型采用 RBAC + 环境维度

## 状态

Accepted

## 背景

平台有三类用户（运维、开发、架构师），需要按角色和环境隔离操作权限。
核心诉求：
- 开发可以自助部署到 dev/test，但不能直接操作 prod
- 运维有全权限
- 架构师可查看全局但不需要操作权限
- prod 环境的部署需要审批

## 决策

采用 RBAC (Role-Based Access Control) 模型，权限矩阵为 `(Role, Service, Environment, Action)` 四维。

角色定义：
- **admin**：平台管理（集群管理、用户管理等）
- **operator**：运维工程师，所有环境的部署和操作权限
- **developer**：开发工程师，dev/test 环境的部署和配置权限，prod 只读
- **viewer**：架构师，所有环境的只读权限

操作类型：deploy, view, config, approve, admin

## 后果

### 正面
- 模型简单清晰，易于理解和实现
- 环境维度天然满足安全隔离需求
- 与审批流程自然结合（prod 需 approve 权限的角色审批）

### 负面
- 不支持按团队/项目维度的细粒度权限（可后续扩展）
- Service 维度的权限管理在 100+ 服务时可能需要按团队批量授权

### 中性
- Permission 表数据量可控（4 角色 × N 服务 × 4 环境 × 5 操作 ≈ 2400N 条）

## 备选方案

| 方案 | 拒绝理由 |
|------|---------|
| ABAC (基于属性) | 过于复杂，当前场景不需要 |
| 无权限控制 | 不满足企业安全要求 |
| 按团队授权 + RBAC | 可作为二期扩展，当前先按角色 |
