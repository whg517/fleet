## Epic 1: 服务接入与生命周期管理

### US-1.1: 新服务接入平台

> **作为** 运维工程师，  
> **我希望** 通过向导式流程将新微服务接入平台，  
> **以便** 服务接入后能通过平台统一管理部署。

**验收条件：**
- 能填写服务基本信息（名称、描述、负责团队、部署类型 K8s/物理节点）
- K8s 服务：关联 Helm Chart 名称和仓库地址，关联 Harbor 项目路径
- 物理节点服务：关联 Ansible Role 名称和仓库地址
- 自动为 4 个环境（dev/test/pre/prod）初始化默认 values override
- 自动创建对应的 Argo CD Application（K8s 服务）
- 接入完成后服务状态为 active
- 全程记录审计日志
- 权限：admin / operator

**关联需求：** D-08, A-03  
**优先级：** P0

---

### US-1.2: 冻结出问题的服务

> **作为** 运维工程师，  
> **我希望** 对某个服务执行冻结操作，  
> **以便** 在紧急止血或维护窗口期间阻止所有新的部署和配置变更。

**验收条件：**
- 冻结操作即时生效
- 冻结后任何人对该服务发起部署或配置变更，请求被拒绝并提示"服务已冻结"
- 进行中的部署不受影响，正常完成
- 冻结/解冻操作记录审计
- 冻结时通过 Webhook 通知相关人
- 权限：admin / operator

**关联需求：** D-07  
**优先级：** P1

---

### US-1.3: 安全下线服务

> **作为** 运维工程师，  
> **我希望** 通过平台安全下线不再需要的服务，  
> **以便** 回收集群资源且不遗漏依赖。

**验收条件：**
- 下线前平台执行校验：最近 7 天是否有流量、是否有待处理部署
- 校验通过后展示确认页面，列出将清理的资源清单
- 确认后执行：删除 K8s 资源、Argo CD Application、相关 ConfigMap/Secret
- 服务状态标记为 offline（保留历史记录，不物理删除）
- 全程记录审计
- 权限：admin / operator

**关联需求：** R-01  
**优先级：** P2

## Epic 2: 构建与版本管理

### US-2.1: Git push 自动触发构建

> **作为** 开发工程师，  
> **我希望** push 代码到指定分支后自动触发构建，  
> **以便** 不需要手动操作就能产出新的镜像版本。

**验收条件：**
- 平台接收 Git Webhook，解析分支和 commit 信息
- 根据服务配置匹配对应的 WorkflowTemplate
- 触发 Argo Workflow 执行构建
- 构建状态在平台 UI 实时可见（pending → running → succeeded/failed）
- 构建完成后自动注册新版本到平台（image ref + git commit + 时间）
- 构建结果通过 Webhook 通知发起人
- 权限：developer（自动触发，无需审批）

**关联需求：** B-01, V-01, N-03  
**优先级：** P1

## Epic 3: 部署管理

### US-3.1: 部署服务到 dev/test（免审批）

> **作为** 开发工程师，  
> **我希望** 选择服务版本部署到 dev 或 test 环境，  
> **以便** 快速验证我的代码变更。

**验收条件：**
- 选择服务 → 选择版本 → 选择环境（dev/test）
- 平台校验：服务未冻结、权限通过、部署锁未占用
- 直接执行部署（无需审批），dev/test auto-sync
- 部署状态实时推送（WebSocket）：syncing → healthy/degraded
- 部署成功/失败通过 Webhook 通知
- 权限：developer（自己的服务）

**关联需求：** D-01, D-03, D-04, N-02  
**优先级：** P0

---

### US-3.2: 部署服务到 pre/prod（需审批）

> **作为** 开发工程师，  
> **我希望** 提交部署请求到 pre/prod 环境，  
> **以便** 经过审批后发布到准生产/生产环境。

**验收条件：**
- 选择服务 → 选择版本 → 选择环境（pre/prod）
- 平台校验：服务未冻结、权限通过、部署锁未占用
- 创建 pending approval 记录
- 通过 Webhook 通知有审批权限的人
- 审批通过后自动执行部署（manual sync）
- 审批超时 24h 自动 reject 并通知提交人
- 部署状态实时推送
- 权限：developer（提交），operator/admin（审批）

**关联需求：** D-01, D-03, D-04, AP-01, AP-02, N-01  
**优先级：** P0

---

### US-3.3: 追踪我的部署审批状态

> **作为** 开发工程师，  
> **我希望** 查看我提交的所有部署请求及其审批状态，  
> **以便** 知道哪些在等待审批、哪些已通过、哪些被拒绝。

**验收条件：**
- 列表展示我提交的部署请求：服务名、环境、版本、提交时间、审批状态
- 支持按状态过滤（pending / approved / rejected / deployed）
- pending 状态的请求可以取消（D-09）
- 被拒绝时展示拒绝原因
- 权限：developer（仅自己的请求）

**关联需求：** AU-02, D-09, AP-01  
**优先级：** P0

---

### US-3.4: 审批部署请求

> **作为** 运维工程师，  
> **我希望** 审批或拒绝提交到 pre/prod 的部署请求，  
> **以便** 控制生产环境发布的质量。

**验收条件：**
- 能查看待审批列表
- 每条审批请求展示：服务名、版本、环境、提交人、提交时间、变更内容摘要
- approve 后自动触发部署
- reject 时填写拒绝原因
- 审批结果通过 Webhook 通知提交人
- 同一请求支持多人查看但只有第一个 approve/reject 生效
- 权限：operator / admin

**关联需求：** AP-01, AP-02, N-01  
**优先级：** P0

---

### US-3.5: 部署回滚

> **作为** 运维工程师，  
> **我希望** 发现部署异常后快速回滚到上一个成功版本，  
> **以便** 最小化线上故障影响。

**验收条件：**
- 手动回滚：一键回滚到上一个 healthy 版本（`argocd app rollback`），< 2min 完成
- 自动回滚：服务配置了自动回滚策略时，Degraded > 5min 自动触发
- prod 自动回滚免审批（紧急恢复优先）
- prod 手动回滚到指定历史版本需审批
- 回滚操作记录审计 + Webhook 通知
- 回滚后平台异步同步 Git 状态
- 权限：operator / admin

**关联需求：** D-05  
**优先级：** P0

---

### US-3.6: 部署物理节点服务

> **作为** 运维工程师，  
> **我希望** 通过平台部署服务到物理节点，  
> **以便** 非容器化服务也能统一管理。

**验收条件：**
- 选择服务 → 选择版本 → 选择目标主机组
- 平台创建 K8s Job 执行 Ansible Runner
- SSH 密钥通过 Secret 挂载，不落磁盘
- 部署状态实时跟踪（Job 状态）
- 部署日志可在平台查看
- 回滚 = 用上一次成功参数重新执行 Ansible
- 权限：operator / admin

**关联需求：** D-02  
**优先级：** P1

---

### US-3.7: 环境晋升

> **作为** 开发工程师，  
> **我希望** 将已验证的版本从 test 晋升到 pre 再到 prod，  
> **以便** 按规范流程逐步发布。

**验收条件：**
- 选择服务和版本，选择目标环境
- 平台校验晋升规则：源环境是否已验证通过、冷却期是否满足
- 晋升到 pre/prod 仍需走审批流程
- 晋升规则可由 admin 配置（必须经过哪个环境、冷却时长）
- 权限：developer（发起），operator（审批）

**关联需求：** PR-01, PR-02  
**优先级：** P1

## Epic 6: 安全与治理

### US-6.1: 通过 SSO 登录平台

> **作为** 任何用户，  
> **我希望** 使用企业 SSO 账号登录平台，  
> **以便** 不需要额外记住一套用户名密码。

**验收条件：**
- 点击登录 → 重定向到企业 SSO（OIDC + PKCE）
- SSO 认证成功后回调平台，建立 session
- Access Token 15min 过期，Refresh Token 7 天，自动刷新
- 支持登出（OIDC RP-Initiated Logout）
- 首次登录自动创建用户记录
- 权限：无（公开端点）

**关联需求：** A-01  
**优先级：** P0

---

### US-6.2: 查看审计日志

> **作为** 运维工程师 / 安全审计员，  
> **我希望** 查询平台所有写操作的审计日志，  
> **以便** 在出问题时追溯谁在什么时候做了什么操作。

**验收条件：**
- 审计日志列表：操作人、操作类型、资源、时间、来源 IP、详情
- 支持多维度过滤：时间范围、操作人、服务、环境、操作类型
- 审计日志不可修改（INSERT-only + hash chain 校验）
- 支持导出（CSV/JSON）
- 权限：admin / auditor

**关联需求：** AU-01, AU-02  
**优先级：** P0

---

### US-6.3: 管理凭证安全

> **作为** 运维工程师，  
> **我希望** 安全地管理平台使用的各种凭证（kubeconfig、Harbor 密码、Git token），  
> **以便** 凭证不被泄露，访问行为可追溯。

**验收条件：**
- 凭证通过 KMS 信封加密存储
- 凭证访问（读取/写入）记录审计
- 凭证不在 API 响文中返回明文
- DEK 定期轮转（30 天）
- 权限：admin（管理），系统自动使用

**关联需求：** SEC-01, SEC-02  
**优先级：** P0

## 需求缺失分析

### 通过用户故事发现的问题

| # | 问题 | 关联故事 | 严重度 | 建议 |
|---|------|---------|--------|------|
| G-1 | **部署前置校验缺失**：部署前没有校验目标版本镜像是否存在于 Harbor | US-3.1, US-3.2 | 🔴 高 | 部署前应校验 image ref 在 Harbor 中存在，否则 Argo CD 部署会失败 |
| G-2 | **配置变更与部署的锁互斥未在故事中体现** | US-4.1, US-3.1 | 🟡 中 | 已在架构文档中定义，但用户故事需明确配置变更和部署共享锁的行为 |
| G-3 | **Webhook 投递失败的补偿机制未定义** | US-8.1 | 🟡 中 | 用户需要知道通知没收到时怎么办。建议：平台内保留通知记录可查询 |
| G-4 | **用户偏好/通知订阅缺失** | US-8.1 | 🟡 中 | 当前 Webhook 是全局配置，没有"用户选择订阅哪些事件"的能力。100+ 服务时通知会爆炸 |
| G-5 | **服务搜索和分组缺失** | US-5.1 | 🟡 中 | 100+ 服务没有搜索、按团队分组、按标签过滤的能力，用户找不到服务 |
| G-6 | **部署历史查询缺失** | US-3.1 | 🟡 中 | 缺少查询某服务在某环境的部署历史记录（谁/什么时候/什么版本/结果） |
| G-7 | **OIDC 用户首次登录的权限初始化缺失** | US-6.1 | 🟡 中 | 首次登录自动创建用户，但默认角色是什么？没有操作权限的用户登录后看到什么？ |
| G-8 | **平台系统配置管理缺失** | — | 🟢 低 | OIDC 配置、Argo CD 地址、Harbor 地址、Git 仓库地址等系统级配置谁来管理？需要系统设置页面 |
