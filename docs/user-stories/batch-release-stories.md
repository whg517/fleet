## 写作说明

本文档覆盖**多服务关联发布**场景。100+ 微服务之间存在依赖关系，经常需要多个服务同时发布或按顺序发布。MVP 阶段先实现批量并行部署，后续迭代支持编排式发布（含依赖顺序、前置条件等）。故事编号以 `BR` 前缀区分，关联 REQUIREMENTS.md 中的需求 ID。

## A. 批量部署

### BR-A.1: 批量部署多个服务到 dev 环境进行联调

> **作为** 开发工程师，  
> **我希望** 一次性选择多个关联服务批量部署到 dev 环境，  
> **以便** 快速完成联调环境搭建，不用逐个服务重复操作。

**场景描述：**
交易团队这周改了 3 个关联服务：order-service（订单服务）、payment-service（支付服务）、inventory-service（库存服务），涉及一个跨服务的优惠券分摊功能。开发完成后需要部署到 dev 环境联调。dev 环境是 auto-sync 的，但需要确认 3 个服务的版本都切到最新。如果逐个部署、逐个等待，来回切页面很麻烦。

**操作步骤：**
1. 登录平台，进入「部署中心」→ 点击「批量部署」
2. 选择目标环境：dev
3. 添加服务批次：
   - order-service → v2.15.0
   - payment-service → v3.2.1
   - inventory-service → v1.8.0
4. 平台对每个服务执行前置校验：镜像存在、Chart 可用（D-10）
5. 平台展示批量部署预览：3 个服务的校验结果、当前版本 vs 目标版本对比
6. 校验全部通过，点击「确认部署」
7. 平台**并行触发** 3 个服务的部署（dev 免审批，auto-sync）
8. 平台展示批量部署进度看板：
   - order-service：✅ 已完成
   - payment-service：🔄 部署中（syncing）
   - inventory-service：✅ 已完成
9. 全部完成后，Webhook 通知发起人
10. 开发开始联调

**验收条件：**
- 支持选择多个服务 + 版本 + 统一目标环境的批量部署（D-01）
- 每个服务独立执行前置校验，有任一校验失败时标注该服务，其他服务不受影响
- 批量部署中每个服务的部署状态独立跟踪（D-04）
- 支持并行触发，不限串行（dev/test 环境）
- 批量部署进度看板一目了然：每个服务的部署状态实时更新
- 部署结果通过 Webhook 通知发起人（N-02）
- 全程记录审计日志，包含批次 ID 关联（AU-01）
- 权限：developer（dev/test 环境），operator/admin（所有环境）

**异常场景：**
- **部分服务前置校验失败**：平台标注失败服务及原因，允许只部署校验通过的服务或取消整个批次
- **部分服务部署失败**：不自动回滚已成功的服务（dev 环境容错性强），平台展示成功/失败汇总，开发者逐个处理失败项
- **同一服务已在部署中**（部署锁冲突）：平台提示该服务正在部署中（D-04），可跳过或等待完成后重试
- **批量部署期间有人单独部署了其中一个服务**：部署锁冲突保护，平台提示该服务状态变化，确认是否继续

**关联需求：** D-01, D-03, D-04, D-10, N-02, AU-01
**优先级：** P1

### BR-A.2: 多服务关联发布到 prod（含审批与协同回滚）

> **作为** 运维工程师，  
> **我希望** 将多个关联服务作为一个发布批次提交到 prod 环境，统一审批、统一监控，异常时一键协同回滚，  
> **以便** 保证关联服务版本一致性，避免部分发布成功部分失败导致版本不兼容。

**场景描述：**
周二发布日，交易团队需要发布优惠券分摊功能，涉及 3 个服务：order-service v2.15.0、payment-service v3.2.1、inventory-service v1.8.0。这 3 个服务必须同时上线——如果只发了 order-service 没发 payment-service，旧版支付服务不认识新版订单服务传来的优惠券字段，会导致支付失败。需要作为一个发布批次提交审批，部署时统一执行，异常时一键全部回滚。

**操作步骤：**
1. 登录平台，进入「部署中心」→「批量部署」→ 选择环境 prod
2. 添加发布批次：
   - order-service → v2.15.0（pre 验证通过 ✅）
   - payment-service → v3.2.1（pre 验证通过 ✅）
   - inventory-service → v1.8.0（pre 验证通过 ✅）
3. 平台执行批量前置校验：
   - 每个服务：镜像存在、Chart 可用（D-10）
   - 晋升规则校验：所有服务 pre 环境验证通过、冷却期满足（PR-02）
   - **关联性校验**：平台检查批次内是否有已知依赖关系，提示用户确认发布顺序
4. 填写统一发布说明（关联 Jira / 变更单号）
5. 提交批量部署请求 → 平台创建**批次审批**（运维主管一次性审批整个批次）
6. 平台通过 Webhook 通知审批人，审批页面展示：
   - 批次内所有服务的变更摘要（版本 diff、pre 环境验证结果、风险评估面板）
   - 关联性提示（"这 3 个服务有关联，需同时上线"）
7. 运维主管审批 approve
8. 平台**并行触发** 3 个服务的部署，实时推送每个服务的部署状态
9. 部署进度看板：
   - order-service：✅ Healthy
   - payment-service：🔄 Syncing
   - inventory-service：✅ Healthy
10. 全部 Healthy 后，平台汇总发布结果，Webhook 通知

**验收条件：**
- 支持多服务批量提交到 prod 环境，统一审批（AP-01）
- 审批页面聚合展示批次内所有服务的变更摘要和风险评估（AP-03）
- 批次内所有服务通过前置校验和晋升规则后才可提交审批（D-10, PR-02）
- 并行触发部署，每个服务状态独立跟踪（D-04）
- 批量部署进度看板实时更新
- 部署结果通过 Webhook 通知（N-02）
- 全程记录审计日志，包含批次 ID 关联（AU-01）
- 权限：operator/admin（提交 prod 批量部署）

**异常场景：**
- **审批被拒绝**：整个批次取消，所有服务不部署，通知发起人
- **部分服务部署失败**：见 BR-A.3 协同回滚场景
- **审批超时**（24h）：整个批次自动 reject，通知发起人（AP-01）
- **批次内某服务被冻结**：前置校验失败，提示具体被冻结的服务，可选择移除该服务或取消批次
- **发布窗口外发起**：平台提示不在发布窗口（D-12），需 admin 特批
- **批次中某服务已有 pending 审批**：平台提示冲突，避免重复提交

**关联需求：** D-01, D-03, D-04, D-10, D-12, AP-01, AP-03, PR-02, N-02, AU-01
**优先级：** P0

## B. 批量回滚

### BR-B.1: 批量回滚关联服务

> **作为** 运维工程师，  
> **我希望** 发布批次中部分服务失败或上线后异常时，能一键回滚整个批次的所有服务，  
> **以便** 快速恢复到发布前的稳定状态，保证关联服务版本一致性。

**场景描述：**
上面的发布批次（order-service v2.15.0 + payment-service v3.2.1 + inventory-service v1.8.0）部署到 prod 后，order-service 和 inventory-service 都 Healthy 了，但 payment-service 部署后错误率飙升到 15%。3 个服务版本必须一致，如果只回滚 payment-service，旧版支付服务不兼容新版订单服务，问题更严重。需要一键回滚整个批次到上一个稳定版本。

**操作步骤：**
1. 收到告警：payment-service prod 错误率飙升（AlertManager → 飞书群）
2. 登录平台，进入发布批次详情页，查看批次状态：
   - order-service：✅ Healthy（但新版本已生效）
   - payment-service：❌ Degraded（错误率 15%）
   - inventory-service：✅ Healthy（但新版本已生效）
3. 点击「**批次回滚**」→ 平台展示回滚预览：
   - order-service：v2.15.0 → v2.14.3（上一个 healthy）
   - payment-service：v3.2.1 → v3.1.2（上一个 healthy）
   - inventory-service：v1.8.0 → v1.7.5（上一个 healthy）
4. 确认回滚 → 平台**并行执行** 3 个服务的回滚
5. 回滚进度看板实时更新：
   - order-service：✅ Rolled back
   - payment-service：✅ Rolled back
   - inventory-service：✅ Rolled back
6. 全部回滚完成后，观察服务健康状态和错误率恢复
7. 回滚操作通过 Webhook 通知团队
8. 事后在发布批次详情查看完整操作记录

**验收条件：**
- 发布批次支持一键回滚所有服务到各自的上一个 healthy 版本（D-05）
- 批次回滚 prod 免审批（紧急止血，同单服务回滚策略）
- 回滚前展示回滚预览：每个服务的回滚版本和 diff
- 并行执行回滚，每个服务回滚状态独立跟踪（D-04）
- 回滚也可选择**指定版本**回滚（非上一个 healthy），此时需走审批
- 回滚操作记录审计日志，关联批次 ID（AU-01）
- 回滚结果通过 Webhook 通知（N-02）
- 如批次中某服务回滚失败，平台标记并提示运维单独处理
- 权限：operator/admin

**异常场景：**
- **部分服务回滚失败**：平台标记回滚失败的服务，提示运维。已成功回滚的服务保持回滚后版本。运维需单独处理失败项（手动回滚 / 检查 Argo CD 状态）
- **回滚目标版本的镜像已被清理**：回滚失败，提示具体服务和镜像缺失。运维需找到可用的历史版本重新指定
- **回滚涉及 DB schema 不兼容**：平台提示风险警告，建议联系开发团队和 DBA 确认兼容性后再回滚
- **同一批次频繁回滚**：平台提示"该批次关联服务 24h 内回滚 N 次"，建议暂停发布排查
- **批次中某服务正在被其他人单独操作**（部署锁冲突）：平台提示锁状态，运维协调后重试
- **自动回滚策略已触发单服务回滚**：如果 payment-service 配置了自动回滚（Degraded > 5min），可能已单独回滚。批次回滚时跳过已回滚的服务，回滚剩余服务

**关联需求：** D-04, D-05, D-11, N-02, AU-01
**优先级：** P0

## C. 编排发布（P2 — 后续迭代）

### BR-C.1: 按依赖顺序编排发布到 prod

> **作为** 运维工程师，  
> **我希望** 按服务依赖关系编排发布顺序（先发底层服务、再发上层服务），并在每一步设置自动校验，  
> **以便** 避免因发布顺序错误导致的服务不兼容和线上故障。

**场景描述：**
一个大型功能上线涉及 4 个服务：基础服务（base-service）、订单服务（order-service）、支付服务（payment-service）、前端网关（api-gateway）。正确的发布顺序应该是：先发 base-service（底层）→ 验证健康 → 发 order-service 和 payment-service（中间层，可并行）→ 验证健康 → 发 api-gateway（上层）。如果顺序反了（比如先发 api-gateway，新版路由指向还没发布的 order-service），网关会报 502。MVP 阶段先做批量并行（BR-A.2），后续迭代支持这种编排式发布。

**操作步骤：**
1. 登录平台，进入「部署中心」→「编排发布」→ 新建发布编排
2. 定义发布流程：
   - **阶段 1**：部署 base-service v2.0.0 → 等待 Healthy → 自动校验（健康检查 + 冒烟测试）
   - **阶段 2**（并行）：部署 order-service v2.15.0 + payment-service v3.2.1 → 等待全部 Healthy → 自动校验
   - **阶段 3**：部署 api-gateway v1.5.0 → 等待 Healthy → 自动校验
3. 配置每阶段的自动校验规则：
   - 基础校验：Argo CD Application 状态为 Healthy
   - 自定义校验：Prometheus 指标阈值（如错误率 < 1%）
   - 人工确认：某阶段后需人工 approve 才继续（可选）
4. 配置异常处理策略：
   - 阶段失败后行为：自动回滚本阶段 + 已完成阶段（全量回滚）/ 仅暂停等待人工处理
5. 提交发布编排 → 走审批流程
6. 审批通过后平台按编排自动执行，实时展示当前阶段和进度
7. 全部阶段完成后通知发布完成

**验收条件：**
- 支持定义多阶段发布流程，每阶段可包含一个或多个服务（D-01）
- 阶段间支持串行（等前序阶段完成）和并行（同一阶段内多服务同时部署）
- 每阶段可配置前置校验条件（健康检查、指标阈值、人工确认）
- 阶段校验失败时按策略执行：自动回滚 / 暂停等待人工处理
- 实时展示编排执行进度（当前阶段、每阶段状态、整体进度）
- 编排发布记录完整审计链路（AU-01）
- 支持保存编排模板，复用到下次发布
- 权限：operator/admin

**异常场景：**
- **某阶段部署后自动校验不通过**（如错误率 > 1%）：按配置策略处理——自动回滚已完成阶段或暂停等待人工。平台展示校验失败详情
- **人工确认环节超时**：暂停等待人工确认的阶段，超时时间可配置（默认 30min），超时后自动回滚已完成阶段
- **阶段间依赖的服务版本不兼容**：平台在编排校验阶段检测版本依赖关系，不满足时提示
- **编排执行过程中平台重启**：编排状态持久化，平台恢复后可继续或人工恢复
- **回滚本身失败**：平台标记回滚失败的服务，告警通知运维紧急介入

**关联需求：** D-01, D-04, D-05, D-06, AP-01, AP-03, N-02, AU-01
**优先级：** P2
