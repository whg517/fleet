# 开发工程师用户故事 — Fleet

> **角色画像：** 后端开发工程师，5 年经验，维护 3-5 个 Go/Java 微服务，不熟悉 K8s/Argo CD，日常在 dev/test 环境反复迭代，每周至少一次生产发布。

---

## A. 代码提交与构建

### US-A.1: push 代码后自动构建镜像
> **作为** 开发工程师，  
> **我希望** 把代码 push 到指定分支后系统自动触发镜像构建，  
> **以便** 我不需要手动去找运维构建镜像，也不用在群里 @ 人。

**场景描述：**
以前每次提交代码后，我需要在飞书群里 @DevOps 同事："帮构建一下 user-service 的 feature-pay 分支"。运维同事忙的时候可能等半小时才看到消息，有时候构建参数变了还要反复沟通。更痛的是下班后或周末改 bug，根本找不到人构建。

**操作步骤：**
1. 开发在本地完成代码，`git push origin feature-pay`
2. 平台检测到 push 事件，自动触发 user-service 的构建流水线
3. 平台从 Git 仓库拉取代码，执行 Dockerfile 构建
4. 构建完成后平台推送镜像到镜像仓库，并通知开发

**验收条件：**
- push 到 `develop`/`feature-*` 分支后 30 秒内自动触发构建
- 构建页面实时显示当前构建阶段（拉代码 → 编译 → 打镜像 → 推镜像）
- 构建结果（成功/失败）通过飞书/钉钉通知提交人
- 构建产物包含 commit hash、分支名、构建时间等元信息

**异常场景：**
- 如果 Git 仓库不可达，平台应明确提示"无法拉取代码"而非长时间卡住
- 如果构建超时（超过 15 分钟），自动标记失败并通知，同时保留已有日志

**关联需求：** REQ-A.1, REQ-A.2
**优先级：** P0

---

### US-A.2: 构建失败后快速定位原因
> **作为** 开发工程师，  
> **我希望** 构建失败时能直接看到失败的步骤和对应的错误日志，  
> **以便** 我不用翻几千行日志去找报错，能快速修了重新提交。

**场景描述：**
之前用 Jenkins 构建失败，我点进去看到的是一大坨黑白日志，编译错误藏在 3000 行输出的中间。有时候是依赖下载超时，有时候是单测失败，但我得自己从头翻到尾。新人更崩溃——根本不知道哪个步骤失败了。

**操作步骤：**
1. 开发收到构建失败通知，点击通知中的链接跳转到构建详情
2. 平台展示构建流水线视图，失败的步骤标红高亮
3. 开发点击失败步骤，查看该步骤的详细日志
4. 平台在日志顶部标注关键错误行（如 `undefined: PaymentService`）
5. 开发根据错误修改代码重新提交

**验收条件：**
- 流水线视图按步骤分阶段展示（拉代码/编译/单测/打镜像/推镜像），每步独立显示成功/失败
- 失败步骤自动滚动到错误位置，高亮 ERROR/FAIL 关键字
- 支持日志关键字搜索（Ctrl+F 搜索框）
- 常见错误类型（编译失败、单测失败、依赖下载超时、Dockerfile 语法错误）给出分类提示
- 构建日志至少保留 30 天

**异常场景：**
- 如果日志量过大（超过 50MB），支持分段加载，不一次性丢全部
- 如果是基础设施问题（如 Docker daemon 挂了），提示"构建环境异常，请联系平台团队"而非展示一堆 daemon 报错

**关联需求：** REQ-A.2, REQ-A.3
**优先级：** P0

### US-A.3: 手动重新构建某个 commit
> **作为** 开发工程师，  
> **我希望** 能在平台上选择任意分支的某个 commit 手动触发构建，  
> **以便** 构建环境偶发失败时不用为了触发重建去推一个空 commit。

**场景描述：**
有一次构建因为 npm 源临时不可用失败了，代码本身没问题。我需要重新构建，但 CI 只在 push 时触发。我的做法是 `git commit --allow-empty -m "rebuild"` 然后推上去，这污染了 git 历史。更尴尬的是，有时候我想构建一个历史 commit 验证问题，但没有手动触发入口。

**操作步骤：**
1. 开发进入 user-service 的构建页面
2. 点击"手动构建"按钮
3. 选择分支（或粘贴 commit hash）
4. 确认构建参数（使用默认或自定义 Dockerfile target）
5. 点击"开始构建"，平台创建构建任务

**验收条件：**
- 支持按分支名或 commit hash 手动触发构建
- 构建参数可配置（如 build target、build args），但提供合理默认值
- 手动构建和自动构建在构建历史中区分标注
- 同一服务同一分支同时只允许一个构建在跑（排队或提示）

**异常场景：**
- 如果指定的 commit hash 不存在，明确提示"未找到该 commit"
- 如果已有构建在进行中，提示"当前有构建正在进行，预计还需 X 分钟"，支持排队或取消

**关联需求：** REQ-A.4
**优先级：** P1

---

### US-A.4: 查看构建进度和排队情况
> **作为** 开发工程师，  
> **我希望** 提交构建后能实时看到构建跑到哪一步了、前面有没有排队，  
> **以便** 我能判断是继续等还是先去干别的，不用每隔 5 分钟去刷一下。

**场景描述：**
以前构建提交后就是个转圈圈"Building..."，不知道在编译还是在推镜像，也不知道前面排了几个任务。经常出现"我以为马上好，结果等了 20 分钟"的情况，很影响节奏。

**操作步骤：**
1. 开发在构建列表页查看当前进行中的构建
2. 平台展示流水线进度条，标注当前阶段（如"正在编译... (3/5)"）
3. 如果在排队，显示前面排队的任务数和预估等待时间
4. 构建完成后进度条变绿，展示总耗时

**验收条件：**
- 构建进度条按步骤实时更新，每步显示已用时间
- 排队任务显示队列位置和预估等待时间
- 构建历史列表支持按服务名/分支/状态过滤
- 构建完成后显示各步骤耗时分解

**异常场景：**
- 如果构建长时间卡在某一步（超过平均耗时 2 倍），标记为"疑似卡住"并提示
- 如果构建节点故障导致任务丢失，通知用户并允许一键重建

**关联需求：** REQ-A.2, REQ-A.5
**优先级：** P1

## B. 自助部署（dev/test）

### US-B.1: 一键部署新版本到 dev 环境
> **作为** 开发工程师，  
> **我希望** 在平台上选择已构建成功的镜像版本，一键部署到 dev 环境，  
> **以便** 我不用写 K8s YAML、不用 kubectl，5 分钟内就能在 dev 环境验证我的代码。

**场景描述：**
以前部署到 dev 环境需要：找运维要 K8s config → 改 deployment YAML 里的 image tag → `kubectl apply` → 自己 `kubectl get pods` 看有没有起来 → 起不来就 `kubectl describe` 排查。这套流程我学了三遍才勉强记住，而且经常改错 YAML 字段导致部署失败。新人来了基本头两周不敢自己部署。

**操作步骤：**
1. 开发进入 user-service 的部署页面
2. 选择目标环境为"dev"
3. 选择要部署的镜像版本（下拉列表展示最近构建成功的版本）
4. 确认部署配置（默认展示当前配置，可调整副本数等）
5. 点击"部署"按钮
6. 平台展示部署进度（滚动更新状态）

**验收条件：**
- 部署操作不超过 3 次点击（选服务 → 选版本 → 确认部署）
- 部署进度实时显示：新 Pod 创建 → 旧 Pod 下线 → 健康检查通过
- 部署成功后展示新版本的服务访问地址
- dev 环境部署不需要审批，开发可自助操作
- 不需要了解 K8s 概念（Deployment/Pod/Service），UI 用业务语言

**异常场景：**
- 如果新版本健康检查不通过，自动回退到上一版本并通知开发
- 如果镜像拉取失败（如 tag 不存在），明确提示"镜像不存在，请确认构建是否成功"
- 如果 dev 环境资源不足，提示"资源不足，请联系平台团队扩容"并列出瓶颈

**关联需求：** REQ-B.1, REQ-B.5
**优先级：** P0

---

### US-B.2: 部署到 test 环境供 QA 测试
> **作为** 开发工程师，  
> **我希望** dev 环境验证通过后，能一键把同一版本部署到 test 环境，  
> **以便** QA 可以立即开始测试，不用我在群里喊"部署好了你们可以测了"。

**场景描述：**
之前的流程：dev 环境我自己验证完 → 飞书通知 QA → QA 说"帮我部署到 test"→ 我又要走一遍 K8s 部署流程部署到 test → 部署完通知 QA。其实 dev 和 test 的部署配置几乎一模一样，但我得操作两次。有时候 dev 上的配置改动忘记同步到 test，QA 测出来的 bug 在我本地根本复现不了。

**操作步骤：**
1. 开发在 user-service 的部署页面
2. 选择"从 dev 环境提升到 test"（或直接选版本部署到 test）
3. 平台自动带出 dev 环境当前运行的版本和配置
4. 开发确认后点击"部署到 test"
5. 平台执行部署并通知相关 QA 同学

**验收条件：**
- 支持"从 dev 提升"操作，自动继承 dev 环境的版本和配置
- 部署到 test 环境自动通知该服务的 QA 负责人
- test 环境部署也无需审批（或仅需要 QA 确认）
- 部署完成后展示 test 环境的访问地址和版本信息

**异常场景：**
- 如果 dev 和 test 配置有预期差异（如数据库地址不同），平台保留差异不做覆盖，仅提示开发确认
- 如果 test 环境有其他人在测试导致部署冲突，提示"test 环境正在被 XX 使用中，请协调后再部署"

**关联需求：** REQ-B.2
**优先级：** P0

---

### US-B.3: 部署失败后快速定位原因
> **作为** 开发工程师，  
> **我希望** 部署失败时平台直接告诉我失败原因和修复建议，  
> **以便** 我不用去学 `kubectl describe pod` 和看一堆 K8s 事件。

**场景描述：**
部署失败最痛苦了。以前部署失败后我运行 `kubectl get pods`，看到 Pod 是 `CrashLoopBackOff`，但完全不知道这什么意思。然后 `kubectl describe pod` 看到一堆事件，关键信息淹没在里面。最后还是得 @ 运维帮忙看，运维说"你的配置里数据库地址写错了"。一个低级错误浪费了一个小时。

**操作步骤：**
1. 开发在部署页面看到部署状态为"失败"
2. 点击"查看详情"
3. 平台展示失败原因分类：
   - 镜像拉取失败
   - 容器启动失败（启动命令/CrashLoopBackOff）
   - 健康检查失败
   - 配置错误
4. 每个分类下展示具体的错误信息和修复建议
5. 开发根据建议修复后可重新部署

**验收条件：**
- 失败原因用中文描述，翻译 K8s 术语（如 CrashLoopBackOff → "容器启动后立即退出，可能是程序启动失败"）
- 提供常见错误的修复建议清单
- 展示容器最近一次的启动日志（应用日志，不是 K8s 事件）
- 失败的部署自动保留现场 30 分钟供排查，之后清理

**异常场景：**
- 如果是 K8s 集群本身不可用，提示"集群异常，请联系平台团队"而非展示 K8s 系统级错误
- 如果多个 Pod 只有部分失败，分别展示每个 Pod 的状态和错误

**关联需求：** REQ-B.3
**优先级：** P0

---

### US-B.4: 同时部署多个服务做联调
> **作为** 开发工程师，  
> **我希望** 能在一个页面上同时部署 user-service、order-service、payment-service 到 dev 环境，  
> **以便** 我做跨服务联调时不用一个个部署等三次，也不用记哪个服务该用哪个版本。

**场景描述：**
改了一个涉及三个服务的接口联调。我需要分别部署三个服务，每次部署完等它 Ready 再部署下一个，整个过程要 15 分钟。中间如果有一个失败了，其他两个已经部署了但版本不匹配，联调时报奇怪的错。我还得记着每个服务该用哪个分支的版本，头大。

**操作步骤：**
1. 开发进入"批量部署"页面
2. 选择目标环境（dev）
3. 添加要部署的服务（搜索/勾选）
4. 每个服务选择要部署的版本（默认推荐最新构建成功的版本）
5. 点击"一键部署"
6. 平台并行执行部署，展示每个服务的部署进度
7. 全部完成后提示"联调环境就绪"并展示访问入口

**验收条件：**
- 支持一次选择多个服务批量部署
- 每个服务可选择不同版本
- 部署进度按服务分卡片展示，独立跟踪
- 支持保存"服务组"模板，下次联调直接使用
- 任一服务部署失败时，其他服务继续部署，最终汇总展示结果
- 部署完成后展示所有服务的版本对照表

**异常场景：**
- 如果部分服务部署成功部分失败，提示"X 个成功，Y 个失败"，失败的可单独重试
- 如果服务间有依赖顺序（如需要先启动数据库），支持配置部署顺序

**关联需求：** REQ-B.4
**优先级：** P1

## C. 配置调试

### US-C.1: 在线修改配置快速试错
> **作为** 开发工程师，  
> **我希望** 在平台上直接修改服务的配置项（如日志级别、超时时间、功能开关），然后重启服务生效，  
> **以便** 我调试问题改个配置不用改代码→提交→构建→部署走一整圈，30 秒搞定。

**场景描述：**
线上排查问题时想把日志级别从 INFO 改成 DEBUG，流程是：改代码里的配置 → git push → 等构建 → 等部署 → 改回来再走一遍。一个来回至少 20 分钟，有时候改配置代码还会忘记改回来提交了。其实这种配置完全应该用 ConfigMap，但我不会写 ConfigMap 的 YAML。

**操作步骤：**
1. 开发进入 user-service 的配置页面
2. 平台展示当前所有配置项（key-value 形式）
3. 开发修改目标配置项（如 `LOG_LEVEL: INFO → DEBUG`）
4. 点击"保存并重启"
5. 平台保存配置、滚动重启服务使其生效
6. 配置变更记录到变更历史

**验收条件：**
- 配置项以表格形式展示，支持搜索和编辑
- 修改配置后一键重启使其生效，无需重新构建镜像
- 配置变更记录修改人、修改时间、修改前后值
- 敏感配置（密码、密钥）脱敏显示，需要权限才能查看
- 支持按环境独立修改配置（dev/test/prod 配置隔离）

**异常场景：**
- 如果配置值格式不合法（如数字字段填了字符串），实时校验并标红提示
- 如果重启后服务无法启动（配置导致），自动回退到上一份配置
- 如果多个用户同时修改同一配置，后提交者提示"配置已被他人修改，请刷新后重试"

**关联需求：** REQ-C.1, REQ-C.4
**优先级：** P0

---

### US-C.2: 对比不同环境的配置差异
> **作为** 开发工程师，  
> **我希望** 能一键对比 dev 和 prod 环境的配置差异，  
> **以便** 排查"dev 正常 prod 报错"的问题时不用两个环境来回切换找不同。

**场景描述：**
"dev 上没问题，prod 上报错"——每个开发最怕听到这句话。以前排查方法：打开 dev 的配置文件，再打开 prod 的配置文件，肉眼逐行对比。配置项几十上百个，看花了也找不到差异。最后往往是某个 timeout 值 dev 是 30 秒 prod 是 5 秒，或者某个 feature flag dev 开了 prod 没开。

**操作步骤：**
1. 开发进入 user-service 的配置页面
2. 选择"环境对比"功能
3. 选择要对比的两个环境（如 dev vs prod）
4. 平台以 diff 视图展示配置差异：
   - 仅 dev 有的配置项（绿色标记）
   - 仅 prod 有的配置项（红色标记）
   - 两边都有但值不同的配置项（黄色高亮）
5. 支持点击差异项查看上下文

**验收条件：**
- 配置对比以 diff 视图展示，差异项高亮
- 支持任意两个环境之间的对比
- 配置项排序保持一致，方便对照
- 支持导出差异报告（用于分享给团队讨论）
- 敏感字段在 diff 中脱敏但保留差异标记

**异常场景：**
- 如果某环境配置加载失败，提示具体原因而非展示空白对比
- 如果配置项数量超过 200 个，提供分组/搜索功能辅助定位

**关联需求：** REQ-C.2
**优先级：** P1

---

### US-C.3: 配置改错后快速回退
> **作为** 开发工程师，  
> **我希望** 配置改错后能一键回退到上一次正常版本，  
> **以便** 我手抖改错一个参数导致服务挂了时不会手忙脚乱。

**场景描述：**
有一次我把连接池配置改错了，导致服务直接起不来。我慌了——忘记之前正确的值是多少，也不记得我具体改了哪几个字段。最后还是翻飞书聊天记录找到之前的截图。整个过程服务 down 了 20 分钟。如果有配置版本回退，10 秒就能恢复。

**操作步骤：**
1. 开发进入 user-service 的配置变更历史
2. 平台展示配置变更列表（时间、修改人、变更摘要）
3. 开发找到出问题前的版本
4. 点击"回退到此版本"
5. 平台展示当前配置和目标配置的差异确认
6. 确认后平台应用旧配置并重启

**验收条件：**

- 回退操作前展示 diff 确认页面
- 回退操作本身也记录在变更历史中（标注"回退操作"）
- 支持回退到任意历史版本，不限最近一次

**异常场景：**
- 如果回退后的配置因环境变化不兼容，重启失败时提示"回退后启动失败，建议手动修复"

**关联需求：** REQ-C.3
**优先级：** P0

## D. 发布到生产

### US-D.1: 提交生产发布申请
> **作为** 开发工程师，  
> **我希望** 在平台上填写发布申请（选服务、选版本、写变更说明），一键提交，  
> **以便** 我不用再写飞书文档走发布流程，也不用追着审批人签字。

以前发版要写一个飞书文档：什么服务、什么版本、改了什么、影响范围、回滚方案，然后发到群里等 @主管 审批，再 @ 运维 执行。文档经常忘记更新（复制上次改改），审批人在开会就干等着。发了之后也说不清审批到哪一步了，只能追着问"批了吗批了吗"。

**操作步骤：**
1. 开发进入"发布管理"页面
2. 选择要发布的服务和版本（从已构建成功的版本中选择）
3. 平台自动带出该版本的 changelog（从 git commit message 汇总）
4. 开发补充发布说明（变更内容、影响范围、回滚方案）
5. 选择发布策略（全量/灰度）和发布时间
6. 提交发布申请
7. 平台自动通知审批人

**验收条件：**
- 发布申请表单必填字段：服务、版本、发布说明、回滚方案

- 提交后自动根据服务确定审批人（对应团队 TL/运维负责人）
- 发布申请创建后生成唯一编号，可在飞书群推送通知
- 支持一次申请发布多个关联服务（关联发布）

**异常场景：**
- 如果选择的版本未在 test 环境验证过，警告"该版本未经过 test 环境验证，确认要发布？"
- 如果同一天已有该服务的发布计划，提示"该服务今日已有发布计划，请确认是否合并"
- 如果发布窗口限制（如重大节假日封网），拒绝提交并解释原因

**关联需求：** REQ-D.1
**优先级：** P0

**异常场景：**
- 如果审批人请假/离职，支持转交或由备选审批人处理

### US-D.3: 发布后发现问题紧急回滚
> **作为** 开发工程师，  
> **我希望** 生产发布后发现异常时能一键回滚到上一个稳定版本，  
> **以便** 我不用在凌晨故障时刻打电话找运维手动操作，减少故障持续时间。

**场景描述：**
有一次晚上 10 点发版，发布后 5 分钟发现有个严重的分页 bug。当时要回滚需要：登录跳板机 → kubectl set image deployment/... → 等滚动更新 → 确认回滚成功。凌晨手抖命令打错了更慌。整个回滚花了 15 分钟，如果有一键回滚可能 2 分钟搞定。

**操作步骤：**
1. 开发在生产发布页面看到部署异常或收到告警
2. 点击"紧急回滚"按钮
3. 平台展示回滚目标版本（默认上一个稳定版本）和回滚影响说明
4. 开发确认回滚
5. 平台执行回滚并通知相关人员
6. 回滚完成后展示当前运行版本和状态

**验收条件：**
- 回滚操作不超过 3 次点击（进入发布详情 → 点击回滚 → 确认）
- 回滚默认指向最近一次健康的版本，也支持选择更早的版本
- 回滚过程实时显示进度
- 回滚操作需要记录操作人、时间、原因（简短填写）
- 回滚后自动通知服务负责人、TL 和运维
- prod 环境回滚需 operator/admin 权限，开发可发起回滚请求

**异常场景：**
- 如果回滚目标版本镜像已被清理（镜像仓库策略），提示"该版本镜像不可用，请选择其他版本"
- 如果回滚过程中数据库 schema 不兼容（新版本做了 schema 变更），警告"检测到数据库结构变更，回滚可能导致数据问题，建议联系 DBA"
- 如果回滚失败，展示失败原因并提示"请联系平台团队手动处理"

**权限说明：**
- dev/test 环境：开发可自助回滚
- pre/prod 环境：开发可发起回滚请求，需 operator/admin 审批后执行（紧急情况下 operator/admin 可直接回滚）

**关联需求：** REQ-D.4
**优先级：** P0

**场景描述：**
排查 bug 最频繁的操作就是看日志。以前看日志的方式：让运维给 SSH 权限 → 登录 Pod → `kubectl logs -f xxx --tail=1000 | grep ERROR`。有时候 Pod 重启了之前的日志就丢了。ELK 倒是有，但查询语法太复杂（KQL/Lucene），每次都要现查怎么写查询。最关键的是：我只看我自己服务的日志，但在 ELK 里筛选服务名都是个门槛。

**操作步骤：**
1. 开发进入 user-service 的日志页面
2. 选择环境（dev/test/prod）和时间范围
3. 输入关键字搜索（如 "timeout"、"panic"、某个订单号）
4. 平台展示匹配的日志列表，支持展开查看完整日志行
5. 支持按日志级别过滤（INFO/WARN/ERROR）
6. 支持"追踪某个 Request ID"查看完整调用链

**验收条件：**
- 日志搜索框支持自然语言关键字搜索，无需学查询语法
- 日志保留至少 7 天（prod 环境保留 30 天）
- 支持按时间范围、日志级别、关键字组合过滤
- 支持按 Request ID/Trace ID 查看完整链路日志
- 日志支持展开查看上下文（前后若干行）
- 日志支持导出（用于提工单或分享）

**异常场景：**
- 如果日志系统不可用，提示"日志服务暂时不可用"而非超时报错
- 如果搜索结果超过 1000 条，分页展示并提供"缩小时间范围"的建议
- 如果该服务没有日志输出（日志级别过高），提示"未找到日志，请检查服务是否在输出日志"

**关联需求：** REQ-E.2
**优先级：** P0

### US-E.3: 查看服务资源使用情况

**异常场景：**
- 如果监控数据采集未配置，提示"该服务未接入监控，请联系平台团队"而非展示空白

## F. 版本管理

### US-F.1: 查看服务各环境运行的版本

**操作步骤：**

   | 环境 | 当前版本 | 部署时间 | 部署人 | 状态 |
   | dev | v1.2.3 (abc1234) | 2024-01-15 14:30 | 张三 | 运行中 |
   | test | v1.2.1 (def5678) | 2024-01-14 10:00 | 李四 | 运行中 |
   | prod | v1.1.9 (ghi9012) | 2024-01-10 16:00 | 王五 | 运行中 |

- 部署人和部署时间明确标注
- 版本不一致时高亮提醒（如"dev 领先 prod 4 个版本"）
- 支持多服务批量查看（我的所有服务的版本矩阵）

**异常场景：**
- 如果某环境服务未部署，显示"未部署"而非空白
- 如果环境数据获取失败，该行标灰并提示重试

**关联需求：** REQ-F.1
**优先级：** P0

**异常场景：**

### US-F.3: 查看版本 Changelog
> **作为** 开发工程师，  
> **我希望** 选择两个版本后能直接看到它们之间的变更内容，  
> **以便** code review、排查问题、写发布说明时不用去 Git 里一个个 commit 翻。

**场景描述：**
写发布说明时需要列出"从 v1.1.9 到 v1.2.3 改了什么"。以前的做法：打开 GitLab，比较两个 tag 的 diff，然后逐个 commit 看 message，手动整理成列表。几十个 commit 混着 merge commit、格式不统一的 message，整理一份发布说明要半小时。QA 也想知道这个版本改了啥好设计测试用例，但看到的也是一堆原始 commit log。

**操作步骤：**
1. 开发进入 user-service 的版本管理页面
2. 选择两个版本（起始版本和目标版本）
3. 平台展示两个版本之间的 changelog：
   - Feature: 新增的功能（从 commit message 中分类）
   - Fix: 修复的问题
   - Refactor: 重构变更
   - Breaking Change: 不兼容变更
4. 支持 commit message 关联的 issue/PR 链接
5. 支持导出 changelog（Markdown 格式）

**验收条件：**
- 选择两个版本后自动生成结构化 changelog
- changelog 按 Feature/Fix/Refactor/Breaking 分类展示
- 每个 commit 显示作者、message、关联的 issue
- 支持导出为 Markdown（可直接粘贴到发布文档）
- merge commit 自动过滤，只展示实际变更

**异常场景：**
- 如果两个版本之间没有变更，提示"无差异"而非展示空白列表
- 如果 commit message 格式不规范（未遵循 Conventional Commits），原始展示但标注"建议规范 commit message"
- 如果选择的起始版本晚于目标版本，自动交换顺序并正常展示

**关联需求：** REQ-F.3
**优先级：** P1
