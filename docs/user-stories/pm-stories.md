## A. 交付管控

### US-A.1: 查看待发布服务队列

> **作为** 项目主管，  
> **我希望** 能在一个视图中看到所有待审批、待发布的服务及其目标环境，  
> **以便** 掌握团队的交付节奏，及时发现堆积和阻塞。

**场景描述：** 周一早会前，我需要快速了解本周有哪些服务在排队等待发布到 pre/prod 环境，哪些审批已经超时，是否有服务被长时间搁置。

**操作步骤：**
1. 登录平台，进入「发布管理 → 待发布队列」
2. 查看待审批、进行中、待发布的任务列表
3. 按团队/环境/优先级过滤
4. 查看每个任务的提交人、提交时间、目标版本、目标环境
5. 标记超时任务（如审批 > 24h 的）

**验收条件：**
- 待发布队列按状态分组展示（待审批 / 审批通过-待执行 / 发布中）
- 显示每个任务的等待时长，超时（> 24h）高亮标识
- 支持按团队、环境维度过滤和分组
- 队列数据实时刷新（或 ≤ 30s 延迟）
- 列表可导出（CSV）供周会使用

**关联需求：** AP-01, D-04, N-01  
**优先级：** P0

---

### US-A.2: 追踪发布进度与阻塞点

> **作为** 项目主管，  
> **我希望** 能实时查看某个服务从构建到部署到各环境的完整流水线状态，  
> **以便** 当发布卡住时快速定位是哪个环节出了问题。

**场景描述：** 某团队反馈"服务 X 发布了一天还没上线"，我需要一眼看出是卡在构建、审批、部署中的哪个环节，并找到责任人。

**操作步骤：**
1. 进入「发布管理 → 发布详情」，选择目标服务
2. 查看流水线视图：构建 → 镜像注册 → dev 部署 → test 部署 → pre 部署 → 审批 → prod 部署
3. 每个环节显示状态（成功/失败/进行中/等待中）和耗时
4. 点击卡住的环节，查看具体原因和当前责任人
5. 可直接从视图跳转到审批操作或日志

**验收条件：**
- 可视化展示端到端发布流水线（构建 → 各环境部署）
- 每个环节标注开始时间、持续时长、当前状态
- 失败/超时的环节醒目标识，附带失败原因摘要
- 支持按单服务和批量（多服务对比）两种视图
- 平台自身构建跟踪（B-01）和部署状态（D-04）数据打通

**关联需求：** B-01, D-04, D-09, D-11  
**优先级：** P0

---

### US-A.3: 发布窗口与冻结期管理

> **作为** 项目主管，  
> **我希望** 能设置发布窗口（如每周二/四 14:00-18:00）和冻结期（如节假日前一周），  
> **以便** 规范团队发布节奏，避免在敏感时段进行高风险变更。

**场景描述：** 国庆长假前一周，需要禁止所有非紧急的 prod 发布。同时日常也希望团队集中在指定窗口发布，便于集中精力保障。

**操作步骤：**
1. 进入「发布管理 → 发布窗口配置」
2. 创建发布窗口规则：指定时间范围、适用环境（仅 prod 或全环境）
3. 创建冻结期规则：指定起止时间、适用范围、允许例外（紧急修复需主管审批）
4. 规则生效后，非窗口期提交的 prod 部署请求自动拒绝或标记为"窗口外"
5. 冻结期内提交的请求自动拦截，例外需主管审批解锁

**验收条件：**
- 支持按环境配置发布窗口和冻结期
- 冻结期内 prod 部署请求被拦截并提示原因
- 支持冻结例外审批流程（紧急 hotfix 走快通道）
- 窗口/冻结规则变更需审计记录
- 通知中心推送窗口/冻结状态变更通知

**关联需求：** D-07, AP-01, N-05  
**优先级：** P1  
⚠️ **需求缺失：** 需求文档中 D-07（服务冻结/解冻）是针对单个服务的冻结，缺少**平台级/全局发布窗口管理**的需求。建议新增「发布窗口与冻结期管理」功能需求。

### US-A.4: 发布风险评估

> **作为** 项目主管，  
> **我希望** 在审批 prod 发布前，能看到该版本的风险评估信息，  
> **以便** 做出更准确的 go/no-go 决策，降低线上事故概率。

**场景描述：** 开发团队提交了一个 prod 发布申请，我作为审批人需要判断：这个版本在低环境充分验证了吗？变更范围大不大？涉及的配置变更多不多？

**操作步骤：**
1. 打开发布审批单，查看风险评估面板
2. 面板展示：该版本在 dev/test/pre 的部署结果和运行时长
3. 面板展示：本版本关联的 commit 数量和 changelog
4. 面板展示：关联的配置变更项列表（values diff）
5. 面板展示：该服务最近 N 次发布的成功率
6. 综合以上信息做出审批决定

**验收条件：**
- 审批页面内嵌风险评估面板，无需跳转多个页面
- 展示低环境验证情况（部署结果 + 运行时长 ≥ 规定时长）
- 展示 changelog（关联 V-02）和配置 diff（关联 C-02）
- 展示该服务历史发布成功率
- 如果跳过 pre 环境直接发 prod，高风险标识

**关联需求：** V-02, C-02, D-04, AP-01, PR-02  
**优先级：** P1  
⚠️ **需求缺失：** 需求文档缺少**发布风险评分/风险摘要**能力。AP-01 只描述了审批流程本身，没有要求审批页提供风险评估信息。建议在审批功能中补充风险评估面板需求。

## B. 服务全局视图

### US-B.1: 查看所有服务的版本分布全景

> **作为** 项目主管，  
> **我希望** 能在一个仪表盘中看到所有服务在四个环境（dev/test/pre/prod）的当前版本，  
> **以便** 快速掌握全局状态，发现版本滞后或不一致的服务。

**场景描述：** 每周部门例会上，我需要展示当前 100+ 服务的版本分布情况，回答"哪些服务 prod 落后于 dev 太多"以及"哪些服务长期没有更新"。

**操作步骤：**
1. 进入「服务管理 → 全局版本看板」
2. 查看矩阵视图：行是服务，列是 dev/test/pre/prod，单元格显示当前版本号
3. 版本不一致的行高亮（如 dev=v2.3 但 prod=v1.8）
4. 支持按团队分组展示
5. 支持按"版本差距"排序，差距最大的排在前面
6. 点击某行可展开该服务的部署历史

**验收条件：**
- 矩阵视图支持 100+ 服务无卡顿
- 版本不一致自动检测和高亮（基于语义化版本比较）
- 支持按团队过滤和版本差距排序
- 点击单元格可查看该版本的部署时间和操作人
- 看板数据延迟 ≤ 5 分钟

**关联需求：** D-04, D-11, SV-01, V-01  
**优先级：** P0

---

### US-B.2: 发现版本不一致与环境间差距

> **作为** 项目主管，  
> **我希望** 系统能自动检测并告警版本不一致的服务（某环境显著落后于其他环境），  
> **以便** 主动推动版本对齐，避免低环境验证的版本和线上版本脱节。

**场景描述：** 某服务在 dev/test 已更新到 v3.0 并通过测试，但 prod 仍在 v2.1，已经滞后 3 周。我需要系统能主动提醒我这种情况。

**操作步骤：**
1. 进入「服务管理 → 版本一致性报告」
2. 系统列出所有版本不一致的服务，标注差距：
   - dev → test：差距几天 / 几个版本
   - test → pre：差距
   - pre → prod：差距
3. 每个不一致项显示：最后晋升时间、阻塞原因（如有审批超时）
4. 支持设置规则：pre→prod 滞后 > 7 天自动告警通知
5. 一键触发晋升流程（将该版本推到下一环境）

**验收条件：**
- 自动检测 dev/test/pre/prod 间的版本差距
- 可配置告警规则（滞后天数阈值、通知渠道）
- 每个不一致项可追溯到原因（审批超时 / 无人提交 / 构建失败等）
- 支持手动触发晋升操作（关联 PR-01）
- 报告可定期自动生成并发送到 Webhook

**关联需求：** PR-01, PR-02, N-05, D-04  
**优先级：** P1  
⚠️ **需求缺失：** 需求文档 PR-01/PR-02 定义了晋升流程，但缺少**版本一致性检测与自动告警**能力。建议补充版本漂移检测需求。

### US-B.3: 资源利用与成本分析

> **作为** 项目主管，  
> **我希望** 能查看各服务占用的集群资源（CPU / 内存）分布和趋势，  
> **以便** 发现资源浪费和优化机会，为容量规划提供数据支撑。

**场景描述：** 季度复盘时，我需要回答"哪些服务占用了最多资源""哪些服务资源配置明显过高"。为下一步的资源优化和成本控制提供依据。

**操作步骤：**
1. 进入「服务管理 → 资源分析」
2. 查看资源排行：按 CPU/内存 request/usage 排序的前 N 个服务
3. 查看资源利用率趋势图（最近 7 天/30 天）
4. 系统标注资源利用率持续偏低的服务（如 CPU 利用率 < 10% 持续 7 天 → 建议缩配）
5. 系统标注资源用量持续增长的服务（可能需要扩容）
6. 按团队汇总资源消耗

**验收条件：**
- 按服务展示 CPU/内存的 request/limit/actual usage
- 资源利用率低的服务自动标注（可配置阈值）
- 资源用量趋势图（至少 7 天）
- 按团队汇总资源消耗总量
- 数据可导出供汇报使用

**关联需求：** M-01, M-02, K-02  
**优先级：** P2  
⚠️ **需求缺失：** 需求文档 K-02 只提供集群级别的资源概览，缺少**服务级资源分析与成本视图**。M-01/M-02 是单个服务的监控，没有聚合分析能力。建议新增服务级资源分析需求。

---

### US-B.4: 技术债识别与老旧服务发现

> **作为** 项目主管，  
> **我希望** 系统能帮我识别长期未更新的服务、使用过时基础镜像的服务、以及配置了过多手动 override 的服务，  
> **以便** 制定技术债偿还计划，避免老旧服务成为隐患。

**场景描述：** 盘点 100+ 服务时，我发现有些服务可能半年以上没有更新，基础镜像可能有安全漏洞。需要一份"技术债清单"来推动改进。

**操作步骤：**
1. 进入「服务管理 → 技术债报告」
2. 查看分类报告：
   - 长期未更新服务（> 90 天无新版本部署）
   - 镜像有已知安全漏洞的服务（关联 Trivy 扫描结果）
   - 配置漂移严重的服务（各环境 values 差异过大）
   - 长期处于降级状态的服务
3. 每项标注技术债等级（高/中/低）
4. 支持按团队分配跟进人
5. 报告可导出并设置定期生成

**验收条件：**
- 自动识别并分类技术债
- 安全漏洞关联 O-02（Trivy 扫描结果）
- 配置漂移检测基于 C-02（配置版本历史）
- 技术债等级可配置
- 报告支持按团队分组和定期推送

**关联需求：** O-02, C-02, D-04, M-01  
**优先级：** P2  
⚠️ **需求缺失：** 需求文档有零散的监控（M-01）和扫描（O-02）能力，但缺少**技术债综合分析与报告**需求。建议新增技术债识别与管理需求。

## C. 团队协作

### US-C.1: 服务归属与负责人确认

> **作为** 项目主管，  
> **我希望** 能快速确认任何一个服务当前归属哪个团队、负责人是谁，  
> **以便** 在跨团队协作或故障响应时快速找到正确的人。

**场景描述：** 线上某服务出现告警，我需要立刻联系负责团队。但 100+ 服务分属多个团队，人员也有变动，我需要可靠的归属信息。

**操作步骤：**
1. 在服务列表或全局搜索中输入服务名
2. 服务详情页展示：Owner 团队、团队负责人、主要开发人员
3. 展示该服务最近活跃的操作人（最近 30 天有操作记录的人员）
4. 点击团队可查看该团队名下所有服务
5. 支持批量查看未归属团队的服务列表（需要分配的）

**验收条件：**
- 每个服务必须关联 Owner 团队（未归属的高亮提醒）
- 展示团队负责人和最近活跃操作人
- 支持按团队筛选查看服务全集
- 归属信息变更需审计记录
- 可导出服务-团队对照表

**关联需求：** A-03, SV-01, AU-01  
**优先级：** P1

---

### US-C.2: 团队权限矩阵查看与审计

> **作为** 项目主管，  
> **我希望** 能查看当前的权限分配矩阵（哪个团队/人员对哪些服务在哪些环境有什么权限），  
> **以便** 确保权限分配合理，避免权限过大或分配遗漏。

**场景描述：** 安全审计时需要确认：是否有离职人员仍有 prod 操作权限？某个新团队成员是否已有 dev 环境权限？权限分配是否有冗余？

**操作步骤：**
1. 进入「系统管理 → 权限矩阵」
2. 选择查看维度：按人员 或 按团队 或 按服务
3. 查看权限明细：人员 × 服务 × 环境 × 操作类型（view/deploy/admin）
4. 筛选异常项：
   - 拥有 prod admin 权限的人员列表
   - 有权限但近 90 天无操作记录的（僵尸权限）
5. 导出权限矩阵供审计

**验收条件：**
- 权限矩阵支持多维度查看（按人/团队/服务）
- 可筛选高权限（prod admin）人员
- 可识别僵尸权限（有权限但长期无操作）
- 矩阵可导出
- 所有权限变更操作有审计记录

**关联需求：** A-02, A-03, A-04, AU-01  
**优先级：** P1  
⚠️ **需求缺失：** 需求文档 A-02 定义了 RBAC 能力，但缺少**权限矩阵可视化查看和僵尸权限检测**需求。建议在审计/权限管理中补充管理视图。

---

### US-C.3: 审批流程效率分析

> **作为** 项目主管，  
> **我希望** 能看到审批流程的效率指标（平均审批时长、超时率、各审批人的响应速度），  
> **以便** 发现审批瓶颈，优化发布流程，减少因审批等待造成的交付延迟。

**场景描述：** 团队抱怨 prod 发布总是卡在审批环节，我需要用数据说话：到底是审批人响应慢，还是提交质量低导致反复打回。

**操作步骤：**
1. 进入「审计与分析 → 审批效率报告」
2. 选择时间范围（最近 7 天/30 天/季度）
3. 查看指标：
   - 平均审批时长（从提交到审批完成）
   - 审批超时率（24h 自动 reject 的占比）
   - 审批通过率 / 打回率
   - 各审批人的平均响应时长
   - 按团队维度的审批效率对比
4. 识别瓶颈：哪个团队/审批人平均耗时最长
5. 导出报告供流程优化讨论

**验收条件：**
- 展示审批平均时长、超时率、通过率/打回率
- 按审批人维度统计响应速度
- 按团队维度对比审批效率
- 可识别审批瓶颈（耗时排名前 N）
- 支持自定义时间范围，可导出

**关联需求：** AP-01, AP-02, AU-01, AU-02  
**优先级：** P2  
⚠️ **需求缺失：** 需求文档 AP-01 定义了审批功能，AU-01/AU-02 定义了审计日志，但缺少**审批效率统计与分析报告**需求。建议在审计模块补充分析能力。

## D. 质量与合规

### US-D.1: 发布频率与成功率统计

> **作为** 项目主管，  
> **我希望** 能查看各团队/服务的发布频率和成功率统计，  
> **以便** 评估团队的交付效率和稳定性，发现需要改进的领域。

**场景描述：** 月度交付总结会上，我需要展示各团队的发布频次和成功率，识别"发布少但失败率高"的团队，帮助他们改进。

**操作步骤：**
1. 进入「审计与分析 → 发布统计」
2. 选择时间范围和维度（按团队/按服务/按环境）
3. 查看指标面板：
   - 发布总次数、成功次数、失败次数

   - 发布频率趋势（按周/月）

4. 团队对比排行：发布频率排名、成功率排名
5. 异常标注：成功率低于阈值（如 < 80%）的团队/服务

**验收条件：**
- 按团队/服务/环境维度统计发布次数和成功率
- 发布频率趋势图（按周/月粒度）
- 团队对比排行榜
- 成功率低于阈值的自动标红
- 支持自定义时间范围，可导出

**关联需求：** D-04, D-11, AU-01, AU-02  
**优先级：** P1  
⚠️ **需求缺失：** 需求文档 AU-02 提供变更追溯查询，但缺少**发布统计聚合分析**能力（按团队/时间维度汇总、趋势、对比）。建议新增发布效能分析需求。

### US-D.2: 回滚率统计与失败分析

**场景描述：** 近期 prod 环境出现了几次紧急回滚，我需要统计哪些服务回滚最频繁，分析是代码质量问题、配置错误还是环境因素。

**操作步骤：**
1. 进入「审计与分析 → 回滚统计」
2. 选择时间范围
3. 查看指标：
   - 回滚总次数、回滚率（回滚次数 / 发布总次数）
   - 自动回滚 vs 手动回滚占比
   - 按服务排序的回滚 TOP 10
   - 按团队汇总的回滚率
4. 查看回滚原因分类（关联告警/构建失败/配置错误等）
5. 点击某次回滚可查看完整的变更链路和根因记录

**验收条件：**
- 统计回滚次数和回滚率，按服务/团队维度展示
- 区分自动回滚和手动回滚
- 回滚原因分类统计
- 可关联到具体的部署记录和审计日志
- 支持时间范围筛选，可导出

**关联需求：** D-05, D-11, AU-01, M-01  
**优先级：** P1  
⚠️ **需求缺失：** 需求文档 D-05 定义了回滚能力，但缺少**回滚率统计与原因分类分析**需求。建议在审计分析模块补充回滚分析能力。

### US-D.3: 审计日志审查与异常检测

> **作为** 项目主管，

> **以便** 及时发现违规操作或异常行为。

**场景描述：** 合规审查要求确认近期所有 prod 环境的变更操作是否合规。我需要能快速筛选出 prod 环境的所有写操作，逐一核查。

**操作步骤：**
1. 进入「审计中心 → 操作日志」
2. 设置筛选条件：环境=prod、时间范围=最近 30 天、操作类型=部署/配置变更/回滚
3. 查看筛选结果列表：操作人、时间、服务、环境、操作类型、操作内容摘要、结果
4. 支持按操作人聚合查看（某人的所有操作）
5. 异常检测：非工作时间操作、批量操作、首次操作某服务等异常标注
6. 审计日志完整性校验：hash chain 验证状态

**验收条件：**
- 支持按人员/时间/服务/环境/操作类型多维度筛选
- 审计日志不可篡改（hash chain 校验，关联 AU-01）
- 异常行为自动标注（非工作时间、批量、首次操作）
- 日志可导出（CSV/JSON）供合规存档
- 查询响应时间 ≤ 3 秒（30 天范围）

**关联需求：** AU-01, AU-02, SEC-02  
**优先级：** P0

## E. 平台运营

### US-E.1: 平台使用情况统计

> **作为** 项目主管，  
> **我希望** 能看到平台自身的使用数据和趋势（接入服务数、活跃用户数、部署次数），  
> **以便** 评估平台推广效果，为后续投入决策提供依据。

**场景描述：** 平台上线运行 3 个月了，老板问"平台用得怎么样"，我需要用数据展示采纳情况和价值。

**操作步骤：**
1. 进入「平台运营 → 使用统计仪表盘」
2. 查看核心指标卡片：
   - 已接入服务总数 / 总服务数（采纳率）
   - 月活用户数（MAU）、周活用户数（WAU）
   - 本月部署总次数、环比增长率
   - 平台 API 调用量趋势
3. 查看趋势图：服务接入数、用户数、部署次数按月趋势
4. 查看团队使用热度排行（哪个团队用得最多/最少）
5. 识别低使用率团队（可能需要培训或推广）

**验收条件：**
- 展示采纳率（已接入/总服务数）
- 展示 MAU/WAU 和部署次数趋势（至少 3 个月）
- 团队使用热度排行
- 低使用率团队自动标注
- 仪表盘可导出截图或 PDF 供汇报

**关联需求：** D-08, SV-01, AU-01  
**优先级：** P2  
⚠️ **需求缺失：** 需求文档缺少**平台运营数据统计**能力。D-08 定义了服务接入流程，但没有接入率、活跃用户、使用趋势等运营分析需求。建议新增平台运营统计需求。

### US-E.2: 新服务接入效率度量

> **作为** 项目主管，  
> **我希望** 能查看新服务从申请接入到首次部署成功的全流程耗时，  
> **以便** 发现接入流程中的瓶颈，降低新服务的入驻成本。

**场景描述：** 有新项目要上 5 个新服务，团队反馈接入流程太长。我需要量化接入耗时，找出哪个环节最慢，推动优化。

**操作步骤：**
1. 进入「平台运营 → 接入效率报告」
2. 查看最近 N 个新服务接入的时间线：
   - Service 注册时间
   - Chart/Role 关联时间
   - Harbor 项目关联时间
   - 环境配置初始化时间
   - 首次成功部署时间
3. 计算总接入耗时（从注册到首次部署成功）
4. 标注每个环节的平均耗时，识别最慢环节
5. 对比历史趋势（接入速度是否在改善）

**验收条件：**
- 展示每个新服务接入的全流程时间线
- 自动计算各环节平均耗时
- 总接入耗时趋势图
- 识别接入最慢的环节
- 支持按团队/时间范围筛选

**关联需求：** D-08, D-01, AU-01  
**优先级：** P2  
⚠️ **需求缺失：** 需求文档 D-08 定义了接入流程的步骤，但缺少**接入效率度量和瓶颈分析**需求。建议补充接入耗时统计。

### US-E.3: 平台健康度监控

> **作为** 项目主管，  
> **我希望** 能查看平台自身及其依赖组件的健康状态，  
> **以便** 在平台故障影响团队工作前主动发现和处理。

**场景描述：** 开发团队反馈部署变慢了，我需要确认是平台本身的问题还是 K8s 集群的问题，各依赖组件运行是否正常。

**操作步骤：**
1. 进入「平台运营 → 平台健康」
2. 查看平台依赖组件状态面板：
   - 数据库（DB）连接状态和响应时间
   - Redis 连接状态
   - Argo CD 连接状态和同步延迟
   - Harbor 连接状态
   - Git 仓库连接状态
   - OIDC/SSO 连接状态
3. 查看平台自身指标：API p95 延迟、错误率
4. 查看历史可用率（SLA 达标情况，目标 99.5%）
5. 异常时展示告警详情和影响范围

**验收条件：**
- 展示所有依赖组件的连接状态和关键指标
- 展示平台 API 性能指标（p95 延迟、错误率）
- 展示 SLA 达标情况（月度可用率）
- 异常状态醒目标识
- 可配置告警通知（组件不健康时推送 Webhook）

**关联需求：** SYS-02, N-04  
**优先级：** P1
