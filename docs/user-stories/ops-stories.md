## A. 日常发布运维

### OPS-A.1: 将版本发布到 pre 环境进行准生产验证

> **作为** 运维工程师，  
> **我希望** 将已在 test 环境验证通过的版本部署到 pre 环境，走完审批流程后发布，  
> **以便** 在生产前做最后一道准环境验证，确保版本质量。

**场景描述：**
周二发布日。订单服务（order-service）在 test 环境跑了两天的回归测试和性能验证，开发团队确认 v2.14.3 可以发布。我需要把这个版本推到 pre 环境，让 QA 做最后一轮准生产验证，确认没问题后再推 prod。pre 环境的部署需要运维审批。

**操作步骤：**
1. 登录平台，在服务列表中搜索 "order-service"
2. 进入 order-service 详情页，确认当前各环境版本分布
3. 点击「部署」→ 选择版本 v2.14.3 → 选择环境 pre
4. 平台执行前置校验：检查镜像是否存在于 Harbor、Chart 版本是否可用
5. 校验通过后，填写发布说明（关联 Jira / 变更单号），提交部署请求
6. 平台创建 pending approval 记录，通过 Webhook 通知审批人（运维主管）
7. 运维主管在平台审批页面查看变更内容摘要，点击 approve
8. 平台自动触发 Argo CD sync，部署状态实时推送（syncing → healthy）
9. 部署成功后，Webhook 通知提交人和相关团队，QA 开始准生产验证

**验收条件：**
- 部署前平台自动校验镜像和 Chart 可用性，不存在则拒绝部署（D-10）
- pre 环境部署必须经过审批，审批超时 24h 自动 reject（AP-01）
- 审批通知通过 Webhook 送达审批人，延迟 < 1min（N-01）
- 部署状态从 syncing 到 healthy 全程实时推送
- 部署成功/失败通过 Webhook 通知发起人（N-02）
- 全程记录审计日志（AU-01）
- 如果服务已被冻结，拒绝部署并提示（D-07）

**异常场景：**
- **镜像不存在**：前置校验失败，平台拒绝提交部署请求，提示具体缺失的镜像 ref
- **审批人未及时审批**：24h 后自动 reject 并通知提交人，提交人可重新发起
- **部署过程中 Argo CD sync 失败**：状态变为 Degraded，平台展示失败原因（Pod events / sync 错误），运维根据情况决定是否回滚
- **审批人同时收到多个发布请求**：平台按提交时间排序展示待审批列表，支持批量审批

**关联需求：** D-01, D-03, D-04, D-10, AP-01, AP-02, N-01, N-02, AU-01
**优先级：** P0

### OPS-A.2: 将验证通过的版本从 pre 晋升到 prod

> **作为** 运维工程师，  
> **我希望** 将 pre 环境验证通过的版本晋升到 prod 环境，  
> **以便** 完成生产环境的正式发布。

**场景描述：**
order-service v2.14.3 在 pre 环境通过了 QA 的准生产验证，现在需要发布到 prod。这是每周二的例行发布。prod 部署需要运维主管审批，并且有发布窗口限制（周二 14:00-18:00）。我需要确认晋升规则满足、发起 prod 部署、审批、监控部署过程。

**操作步骤：**
1. 在平台确认 order-service 在 pre 环境状态为 healthy，且运行时间 > 冷却期（配置为 2h）
2. 点击「环境晋升」→ 选择版本 v2.14.3 → 目标环境 prod
3. 平台校验晋升规则：源环境（pre）已验证通过、冷却期满足、版本一致
4. 校验通过后提交部署请求，平台创建 pending approval
5. 运维主管审批（查看变更摘要、pre 环境验证结果）
6. 审批通过后 Argo CD 执行 sync，实时推送部署状态
7. 部署完成后观察 prod 环境服务健康状态（持续 10-15min）
8. 确认无异常后通知发布完成

**验收条件：**
- 晋升规则可配置化（PR-02）：必须经过 pre 环境验证、冷却期时长
- 不满足晋升规则时拒绝并提示具体原因
- prod 环境部署必须经过审批（AP-01）
- 部署前置校验（D-10）：镜像存在、Chart 可用
- 部署状态实时可见
- Webhook 通知发起人和相关团队（N-02）
- 审计日志完整记录（AU-01）

**异常场景：**
- **晋升规则不满足**（pre 还没验证 / 冷却期不够）：平台拒绝并提示，不可强制跳过（admin 可配置跳过）
- **部署后服务 Degraded 超过 5min**：如果配置了自动回滚策略，自动回滚到上一个 healthy 版本；否则手动触发回滚
- **发布窗口外发起部署**：平台提示当前不在发布窗口内，需 admin 特批
- **部署过程中发现配置遗漏**（如缺少新的环境变量）：暂停当前部署 → 修改 values → 重新部署

**关联需求：** PR-01, PR-02, D-01, D-03, D-04, D-05, D-10, AP-01, N-02, AU-01
**优先级：** P0

### OPS-A.3: 生产环境紧急回滚

> **作为** 运维工程师，  
> **我希望** 发现线上服务异常后一键回滚到上一个稳定版本，  
> **以便** 在最短时间内恢复服务，最小化业务影响。

**场景描述：**
周四晚上 22:00，支付服务（payment-service）发布了 v3.2.0 到 prod。部署后 3 分钟，Prometheus 告警：错误率从 0.1% 飙升到 15%。确认是新版 bug 后，需要立刻回滚到 v3.1.2（上一个稳定版本）。这种情况每秒都是金钱损失，回滚操作必须快速且可靠。

**操作步骤：**
1. 收到告警通知（AlertManager Webhook → 飞书群），确认 payment-service 异常
2. 登录平台，进入 payment-service 详情页，查看当前 prod 状态为 Degraded
3. 点击「回滚」→ 平台展示上一个 healthy 版本（v3.1.2）→ 点击「确认回滚」
4. 平台执行 `argocd app rollback`，prod 自动回滚免审批（紧急恢复优先）
5. 10-30s 内回滚完成，观察错误率是否恢复
6. 回滚操作通过 Webhook 通知团队
7. 事后在平台查看部署历史，记录回滚原因

**验收条件：**
- 一键回滚到上一个 healthy 版本，操作 < 2min 完成（D-05）
- prod 自动回滚免审批（紧急止血优先）
- 回滚到**指定历史版本**（非上一个）需走审批
- 如果配置了自动回滚策略，Degraded > 5min 自动触发回滚（D-05）
- 回滚操作记录审计 + Webhook 通知（AU-01, N-02）
- 回滚后平台异步同步 Git 状态

**异常场景：**
- **回滚本身失败**（如上一个版本的镜像已被清理）：平台报错并展示具体原因，运维需手动找到可用版本回滚
- **自动回滚触发但 Argo CD 不可达**：平台记录回滚失败告警，通知运维人工介入
- **回滚后服务仍然异常**：说明问题可能不在版本变更，需要进入故障排查流程（OPS-B.1）
- **同一服务频繁回滚**：平台告警提示"该服务 24h 内回滚 N 次"，建议暂停发布排查

**关联需求：** D-05, D-11, AU-01, N-02, M-03
**优先级：** P0

### OPS-A.4: 部署卡住在 syncing 状态排查

> **作为** 运维工程师，  
> **我希望** 当部署长时间停留在 syncing 状态时，能快速定位卡住原因并处理，  
> **以便** 避免部署窗口超时或服务长时间处于不稳定状态。

**场景描述：**
周三下午发布用户服务（user-service）v1.8.0 到 prod，Argo CD sync 已触发但 8 分钟过去了状态还在 syncing。100+ 服务的集群中，这种卡住可能是镜像拉取慢、资源不足、PVC 绑定失败、准入 webhook 拒绝等多种原因。我需要平台帮我快速定位问题。

**操作步骤：**
1. 在平台部署列表发现 user-service 状态为 syncing，持续时间 > 5min（平台标黄提示）
2. 点击进入部署详情，查看 Argo CD 返回的 sync 状态和资源事件
3. 平台展示关键诊断信息：
   - Pod 状态分布（Pending/ContainerCreating/CrashLoopBackOff）
   - 最近 Events（ImagePullBackOff、Insufficient cpu、FailedScheduling 等）
   - 节点资源水位（是否某节点资源耗尽）
4. 根据诊断信息定位问题（例如：镜像拉取慢 → 检查 Harbor 状态和网络；资源不足 → 扩容节点或调整 requests）
5. 问题修复后 Argo CD 自动重试 sync，或手动触发 sync
6. 如果无法短时间解决，决定是否回滚

**验收条件：**
- 平台展示 Argo CD sync 状态和持续时间，超过阈值（可配置，默认 5min）标黄提醒（D-04）
- 平台聚合展示相关诊断信息：Pod 状态、Events、节点水位（K-02, K-03）
- 支持查看 Pod 日志快速定位启动失败原因（K-04）
- sync 超时后平台可选择「继续等待」或「回滚」
- 全过程操作记录审计（AU-01）

**异常场景：**
- **Argo CD 自身不可达**：平台检测到 Argo CD 状态异常（SYS-02），提示运维检查 Argo CD 健康
- **多个服务同时卡住**：可能是集群级别问题（节点宕机、网络故障），平台集群概览页可以看到全局异常
- **镜像拉取超时**：平台提示具体超时的镜像 ref，可跳转到 Harbor 确认镜像是否存在

**关联需求：** D-04, K-02, K-03, K-04, SYS-02, AU-01
**优先级：** P1

### OPS-A.5: 节假日前批量冻结发布

> **作为** 运维主管，  
> **我希望** 在法定节假日前批量冻结所有服务的发布，  
> **以便** 避免节假日期间因发布引发故障而无人值守。

**场景描述：**
国庆节前 2 天（9 月 30 日），按照公司发布制度，节假日期间冻结所有非紧急变更。我需要在平台上把 prod 环境的所有服务一键冻结，节后统一解冻。100+ 服务逐个冻结不现实，必须支持批量操作。

**操作步骤：**
1. 登录平台，进入「服务管理」→ 选择环境 prod
2. 使用批量操作「批量冻结」，勾选全部服务（或按团队/标签过滤后全选）
3. 确认冻结操作，填写冻结原因和预计解冻时间
4. 平台批量执行冻结，所有 prod 服务的部署和配置变更被阻止
5. 冻结期间如果有人尝试提交部署，请求被拒绝并提示"节假日冻结期，预计 X 日解冻"
6. 节后（10 月 8 日）使用批量解冻恢复所有服务

**验收条件：**
- 支持批量冻结/解冻操作，按环境、团队、标签维度批量选择（D-07）
- 批量冻结时填写统一冻结原因和预计解冻时间
- 冻结后所有部署和配置变更请求被拒绝，并展示冻结原因（D-07）
- 冻结/解冻操作通过 Webhook 通知所有相关团队（N-02）
- 支持冻结期间紧急解冻单个服务（线上故障修复），需 admin 审批
- 全程记录审计日志（AU-01）

**异常场景：**
- **冻结期间出现紧急线上故障需要发布修复版本**：admin 可单独解冻指定服务 → 紧急发布 → 重新冻结
- **批量操作部分成功部分失败**：平台展示成功/失败列表，失败项可重试
- **忘记解冻**：冻结时设置的预计解冻时间到达后，平台提醒运维主管是否解冻

**关联需求：** D-07, N-02, AU-01
**优先级：** P1

## B. 故障应急

### OPS-B.1: 线上服务突发放障排查

> **作为** 运维工程师，  
> **我希望** 服务出现异常时能在平台快速走完标准排查路径，  
> **以便** 在 15 分钟内定位故障根因并采取行动。

**场景描述：**
周一上午 10:15，告警群突然收到多条告警：商品服务（product-service）在 prod 环境的 P99 延迟从 50ms 飙到 800ms，错误率 5%。同时关联的搜索服务也开始报警。这是大促前夕，影响面很大，需要快速定位是代码 bug、配置变更、依赖中间件故障、还是流量突涨。

**操作步骤：**
1. 登录平台，查看服务健康状态总览，确认 product-service 状态为 Degraded
2. 进入 product-service 详情页，查看：
   - **部署历史**：最近是否有发布/配置变更——确认昨晚 23:00 有人改了 DB 连接池配置
   - **Pod 状态**：是否有重启、OOM、CrashLoopBackOff
   - **Pod 日志**：查看异常 Pod 的 stdout，搜索 error/panic 关键字
   - **监控指标**：CPU/内存/流量曲线，定位异常起始时间
   - **配置变更历史**：diff 昨晚的配置变更，确认连接池参数从 10 改成了 100 导致 DB 被打满
3. 定位根因：配置变更导致 DB 连接打满
4. 操作：回滚配置到上一个版本或直接修改连接池参数紧急修复
5. 修复后观察服务恢复情况

**验收条件：**
- 服务详情页聚合展示排查所需关键信息：部署历史、配置变更历史、Pod 状态、日志、监控指标（M-01, M-02, K-03, K-04, D-11, C-02）
- 告警集成展示（M-03），能从告警直接跳转到服务详情
- 日志查看支持关键字搜索和过滤（K-04）
- 配置变更历史支持 diff 查看（C-02）
- 回滚配置/版本操作 < 2min（D-05, C-03）
- 全程操作记录审计（AU-01）

**异常场景：**
- **平台自身不可用**：运维需直接通过 kubectl / argocd CLI 操作，平时应有应急预案（SOP 文档）
- **多个服务同时故障**：可能是公共依赖故障（DB/Redis/网关），平台全局视图帮助快速定位共同模式
- **日志量太大无法查看**：平台日志限制单次拉取量，支持跳转到 Loki/ES 查看完整日志（M-04）
- **排查过程中被人发起新部署干扰**：先冻结服务再排查（D-07）

**关联需求：** M-01, M-02, M-03, M-04, K-03, K-04, D-11, C-02, D-05, D-07, AU-01
**优先级：** P0

### OPS-B.2: K8s 节点故障处理

> **作为** 运维工程师，  
> **我希望** 节点出现 NotReady 或资源耗尽时能快速处置，  
> **以便** 保证节点上的服务 Pod 能正常调度和运行。

**场景描述：**
周四下午，Prometheus 告警：worker-node-07 磁盘使用率 95%，节点状态不稳定。该节点上跑了 30+ 个 Pod，涉及多个服务。需要快速排查节点问题并做安全处理，避免节点彻底宕机影响业务。

**操作步骤：**
1. 登录平台，进入「集群管理」→ 节点列表
2. 查看 worker-node-07 的状态：NotReady 或 DiskPressure 条件
3. 查看节点详情：CPU/内存/磁盘使用率、运行 Pod 数量、最近 Events
4. 决策处理方式：
   - **情况 A（磁盘问题）**：尝试清理无用镜像/日志 → 观察恢复
   - **情况 B（节点不健康）**：cordon 节点 → drain 节点（安全驱逐 Pod）→ 修复节点 → uncordon
5. 执行 drain 时观察受影响服务的 Pod 重新调度情况
6. 确认所有服务恢复正常

**验收条件：**
- 节点列表展示所有节点的状态、资源使用率、Pod 数量（K-02）
- 异常节点高亮标记（DiskPressure / MemoryPressure / NotReady）
- 支持在平台执行 cordon / drain / uncordon 操作（K-05）
- drain 时平台提示受影响的服务列表和 Pod 数量
- drain 操作需要二次确认（防止误操作）
- 节点操作记录审计（AU-01）
- 权限：operator / admin

**异常场景：**
- **drain 过程中 Pod 无法驱逐**（PDB 限制、local storage）：平台展示具体阻塞原因，运维可强制 drain 或先处理 PDB
- **drain 后 Pod 无法重新调度**（集群资源不足）：平台提示集群资源水位，运维需临时扩容节点或缩容其他服务释放资源
- **节点彻底宕机无法 drain**：平台提示节点不可达，运维需强制删除 Pod 让它们重新调度
- **多个节点同时故障**：平台告警，提示集群级风险，建议立即扩容

**关联需求：** K-02, K-05, K-03, AU-01
**优先级：** P1

### OPS-B.3: 流量突涨时的紧急扩容

> **作为** 运维工程师，  
> **我希望** 服务流量突涨时能在平台快速扩容，  
> **以便** 保证服务在流量高峰期间的可用性。

**场景描述：**
周五晚上 19:00，营销活动开始，订单服务（order-service）流量瞬间涨了 5 倍。当前 4 个副本的 CPU 使用率已经 85%，需要立刻扩容到 12 个副本。但集群总资源是否够用？不够的话怎么处理？必须在 5 分钟内完成扩容。

**操作步骤：**
1. 收到告警：order-service CPU 使用率 > 80%（AlertManager → 飞书群）

5. **情况 B（资源不足）**：先缩容非核心服务释放资源 → 或临时扩容集群节点

7. 活动结束后调整回正常副本数（或恢复 HPA 策略）

**验收条件：**

**异常场景：**

> **作为** 运维工程师，  
> **我希望** 定期轮换平台使用的各种凭证（Git Token、Harbor 密码、kubeconfig），  
> **以便** 满足安全合规要求，降低凭证泄露风险。

**场景描述：**
公司安全制度要求所有凭证每 90 天必须轮换一次。平台使用的关键凭证包括：Git API Token（拉取 Helm Chart 和触发构建）、Harbor 密码（拉取镜像）、kubeconfig（管理 K8s 集群）。100+ 服务依赖这些凭证，轮换必须在不停服的情况下完成。

**操作步骤：**
1. 登录平台，进入「凭证管理」
2. 查看所有凭证列表和上次更新时间：
   - Git Token：上次更新 85 天前（即将到期）
   - Harbor 密码：上次更新 60 天前
   - kubeconfig：上次更新 120 天前（已过期）
3. 选择需要轮换的凭证 → 更新凭证值
4. 平台通过 KMS 信封加密存储新凭证
5. 平台验证新凭证是否有效（尝试用新凭证连接对应服务）
6. 验证通过后，旧凭证失效
7. 平台记录审计日志

**验收条件：**
- 凭证通过 KMS 信封加密存储（SEC-01）
- 凭证更新后自动验证有效性（连接测试）
- 凭证不在 API 响文中返回明文（SEC-01）
- 凭证访问（读取/写入）记录审计（SEC-02）
- 凭证即将过期（< 7 天）时平台告警提醒
- DEK 定期轮转（30 天）
- 权限：admin

**异常场景：**
- **新凭证验证失败**（如 Token 权限不够）：平台拒绝更新，保留旧凭证，提示具体错误
- **轮换后服务异常**（如新 Harbor 密码权限不足导致镜像拉取失败）：快速回退到旧凭证
- **同时轮换多个凭证导致服务中断**：平台支持逐个轮换 + 验证，避免批量操作风险
- **kubeconfig 轮换失败导致平台无法管理集群**：这是严重故障，平台需有备用访问方式

**关联需求：** SEC-01, SEC-02, AU-01
**优先级：** P0

### OPS-E.3: 权限审查

> **作为** 运维主管，  
> **我希望** 定期审查平台用户权限，回收离职人员和调岗人员的权限，  
> **以便** 满足最小权限原则和安全审计要求。

**场景描述：**
每季度安全审计要求审查平台所有用户的权限。需要排查：哪些用户有 prod 环境的 operator 权限、是否有离职人员账号仍然有效、是否有开发人员误获得了 admin 权限。100+ 服务 × 多个环境 × 多种操作类型，权限矩阵很复杂。

**操作步骤：**
1. 登录平台，进入「权限管理」→ 用户列表
2. 查看所有用户及其角色：
   - 用户名、角色（admin/operator/developer/viewer）、所属团队、最近登录时间
3. 筛选最近 30 天未登录的用户（可能是离职或调岗人员）
4. 筛选有 prod 环境操作权限的用户列表
5. 逐个确认：
   - 离职人员 → 回收所有权限（设为 disabled）
   - 调岗人员 → 调整角色（如 operator → viewer）
   - 权限过高 → 降级
6. 导出权限审查报告

**验收条件：**
- 用户列表展示用户名、角色、所属团队、最近登录时间（A-02, A-03）
- 支持按角色、团队过滤
- admin 可调整用户角色（A-04）
- 支持禁用用户（设为 disabled，无法登录）
- 支持导出权限审查报告
- 全程记录审计（AU-01）

**异常场景：**
- **误操作降级了自己的权限**：平台防止 admin 降级自己（至少保留一个 admin）
- **离职人员 SSO 账号仍有效**：平台侧禁用后用户无法登录平台，但 SSO 侧需同步通知 IT 处理
- **权限变更后用户正在操作中**：当前 session 继续有效直到 token 过期（15min），之后新权限生效

**关联需求：** A-01, A-02, A-03, A-04, AU-01
**优先级：** P1

## F. 平台管理

### OPS-F.1: 平台健康自检

> **作为** 运维工程师，  
> **我希望** 在平台首页看到平台自身依赖组件的健康状态，  
> **以便** 在平台故障前发现依赖组件异常，提前处理。

**场景描述：**
平台依赖多个外部组件：PostgreSQL（主 DB）、Redis（缓存）、Argo CD（部署引擎）、Harbor（镜像仓库）、Git 仓库（配置和 Chart）、Prometheus（监控数据）、OIDC Provider（SSO）。任何一个组件故障都可能导致平台部分功能不可用。我需要一眼看到所有依赖的健康状态。

**操作步骤：**
1. 登录平台，首页或「系统设置」→ 「平台健康」
2. 平台展示依赖组件健康状态：
   - PostgreSQL：✅ Healthy（响应时间 3ms）
   - Redis：✅ Healthy（响应时间 1ms）
   - Argo CD：✅ Healthy（API 可达）
   - Harbor：⚠️ Degraded（API 响应慢，5s 超时）
   - Git 仓库：✅ Healthy
   - Prometheus：✅ Healthy
   - OIDC Provider：✅ Healthy
3. 发现 Harbor 异常 → 点击查看详情 → 查看错误日志
4. 联系 Harbor 团队排查，或临时切换到备用 Harbor

**验收条件：**
- 平台首页或系统设置页展示所有依赖组件的健康状态（SYS-02）
- 每个组件展示最近检查时间、响应时间、状态
- 异常组件高亮展示具体错误信息
- 组件异常时通过 Webhook 通知运维团队（N-02）
- 健康检查间隔可配置（默认 30s）
- 权限：所有角色（查看），admin（配置）

**异常场景：**
- **PostgreSQL 故障**：平台大部分功能不可用。平台展示降级提示，运维需紧急恢复 DB。注意：平台故障**不影响线上已部署服务**的运行
- **Argo CD 故障**：部署、回滚、配置变更功能不可用，但服务查看和监控正常。运维需紧急恢复 Argo CD
- **OIDC 故障**：新用户无法登录，已登录用户 token 过期后也无法刷新。运维需紧急恢复 SSO
- **多个组件同时故障**：平台提示全局故障，建议启动应急预案

**关联需求：** SYS-02, N-02
**优先级：** P1

### OPS-F.2: 系统配置变更

> **作为** 运维主管，  
> **我希望** 在平台管理系统级配置（OIDC 连接信息、Argo CD 地址、Harbor 地址等），  
> **以便** 在基础设施变更时快速更新平台配置。

**场景描述：**
公司要迁移 Argo CD 到新的集群，Argo CD 的 API 地址从 `https://argocd.internal.company.com` 变成 `https://argocd-v2.internal.company.com`。同时 Harbor 也要升级，API 版本从 v2.0 到 v2.5。我需要在平台更新这些系统配置，确保平台能正常对接新的基础设施。

**操作步骤：**
1. 登录平台，进入「系统设置」→ 「基础设施配置」
2. 修改 Argo CD 地址为新 URL → 测试连接 → 保存
3. 修改 Harbor 地址和 API 版本 → 测试连接 → 保存
4. 如果修改了 OIDC 配置（如 issuer URL），需要测试 SSO 登录是否正常
5. 更新完成后观察平台各功能是否正常工作
6. 记录变更审计日志

**验收条件：**
- 系统设置页可管理 OIDC、Argo CD、Harbor、Git 等系统级配置（SYS-01）
- 配置修改后支持测试连接（验证配置是否正确）
- 凭证类配置（如 OIDC client secret）通过 KMS 加密存储（SEC-01）
- 配置变更记录审计（AU-01）
- 权限：admin

**异常场景：**
- **配置错误导致连接失败**：平台保留上一份有效配置，支持一键回退
- **修改 OIDC 配置后 SSO 登录失败**：平台保留一个本地 admin 后门账号（仅紧急使用），用于在 SSO 故障时登录修复
- **Argo CD 地址变更后进行中的部署状态丢失**：平台提示运维确认所有部署完成后再切换

**关联需求：** SYS-01, SEC-01, AU-01
**优先级：** P0

### OPS-F.3: Webhook 通知配置与调优

> **作为** 运维主管，  
> **我希望** 配置和管理平台的 Webhook 通知规则，  
> **以便** 团队能及时收到审批、部署、构建等事件通知，且不被通知轰炸。

**场景描述：**
平台上线后，100+ 服务每天产生大量的部署、构建、配置变更事件。如果所有事件都推到飞书群，消息量太大导致重要通知被淹没。我需要精细化配置 Webhook 通知规则：审批通知必须即时送达、部署结果按团队分群通知、构建失败才通知（成功不通知）。

**操作步骤：**
1. 登录平台，进入「系统设置」→ 「Webhook 配置」
2. 创建多个 Webhook 配置：
   - **Webhook 1（运维群）**：URL = 飞书运维群 Webhook
     - 订阅事件：approval（审批请求）、alert（告警转发）
     - 过滤：全部服务，全部环境
   - **Webhook 2（各团队群）**：URL = 飞书各团队群 Webhook
     - 订阅事件：deployment（部署结果）、build（构建结果）
     - 过滤：按服务/团队分组（order-service 的通知只发到交易团队群）
     - 过滤规则：构建只在 failed 时通知，部署 success 和 failed 都通知
3. 配置重试策略：失败重试 3 次，间隔 10s / 30s / 60s
4. 观察通知效果，根据团队
反馈调整过滤规则
5. 在「通知记录」页面查看通知发送历史和成功率

**验收条件：**
- 支持创建多个 Webhook 配置，每个指定 URL、Secret、订阅事件（N-05）
- 支持按服务/环境/团队过滤通知（N-06）
- 支持配置重试次数和重试间隔（N-05）
- 支持启用/禁用单个 Webhook 配置（N-05）
- 平台内保留通知发送记录，可查看历史通知（N-07）
- 支持查看通知发送状态（成功/失败/重试中）
- 权限：admin

**异常场景：**
- **Webhook 端点不可达**：平台按重试策略重试，重试耗尽后标记为失败，在通知记录中可查看
- **Webhook Secret 错误**（签名验证失败）：平台提示签名校验失败，运维需检查 Secret 配置
- **通知量太大导致飞书群限流**：平台支持合并通知（如 5min 内同一服务的多个事件合并为一条）
- **配置了错误的 URL**：平台在保存时尝试发送测试通知，验证 URL 是否可达

**关联需求：** N-05, N-06, N-07, N-01, N-02, N-03, N-04
**优先级：** P0
