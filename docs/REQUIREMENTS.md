## 1. 背景

### 1.1 现状

- 微服务 100+ 且持续增长，全部运行在 K8s 上，个别服务部署在物理机
- 服务部署、配置变更、版本管理主要依赖人工操作或零散脚本
- 缺乏统一的服务生命周期管理入口
- 多环境（dev / test / pre / prod）的发布流程不规范，依赖个人经验
- 操作审计和变更追溯能力薄弱

### 1.2 问题

| 问题域 | 具体痛点 |
|--------|---------|
| 部署效率 | 手工操作多、重复劳动、容易出错 |
| 环境管理 | 环境间配置差异不透明，promoting 困难 |
| 版本追踪 | 不知道某个环境跑的什么版本，回滚靠记忆 |
| 权限控制 | 谁能操作 prod 没有统一管控 |
| 审计合规 | 操作无记录，出问题难追溯 |
| 服务可见性 | 架构师看不到全局拓扑和版本分布 |

### 1.3 项目目标

构建统一的 Fleet 平台，覆盖服务完整生命周期：

```
构建打包 → 版本管理 → 多环境部署 → 配置变更 → 监控 → 扩缩容 → 服务下线
```

同时提供集群管理、OCI 管理、认证权限、审计、发布审批等平台能力。

## 2. 干系人

| 角色 | 核心诉求 | 使用场景 |
|------|---------|---------|
| **运维工程师** | 全权限管理所有服务和基础设施 | 集群纳管、部署操作、故障排查、资源治理 |
| **开发工程师** | 自助管理自己的服务 | 部署到 dev/test、查看日志和监控、修改配置 |
| **架构师** | 全局视角掌握系统状态 | 查看版本分布、资源利用、技术债识别 |

## 3. 功能需求

### 3.1 构建打包

| ID | 功能项 | 优先级 | 描述 |
|----|--------|--------|------|
| B-01 | CI 触发与跟踪 | P1 | 对接 Argo Workflows，支持两种触发方式：Git push 自动触发（Webhook）和平台 UI 手动触发。平台跟踪构建状态 |
| B-02 | 构建模板管理 | P1 | 管理 WorkflowTemplate OCI 制品，支持平台创建发布和外部 OCI 注册，参数化触发 |
| B-03 | 构建日志查询 | P1 | 查看构建过程日志 |

### 3.2 版本管理

| ID | 功能项 | 优先级 | 描述 |
|----|--------|--------|------|
| V-01 | 镜像版本注册 | P1 | 构建完成后自动注册到平台，MVP 阶段支持手动注册/API 导入 |
| V-02 | 版本历史与 changelog | P1 | 关联 git commit，记录变更内容 |
| V-03 | 版本对比 | P2 | 对比两个版本的差异 |

### 3.3 多环境部署

| ID | 功能项 | 优先级 | 描述 |
|----|--------|--------|------|
| D-01 | Helm 部署（K8s） | P0 | 通过 Argo CD Application 管理 Helm Chart 部署 |
| D-02 | Ansible 部署（物理节点） | P1 | 通过 Ansible Role 模板部署服务到物理节点，平台统一触发和跟踪 |
| D-03 | 环境配置 override | P0 | 每个环境独立的 values override |
| D-04 | 部署状态跟踪 | P0 | 实时获取 Argo CD sync 状态 |
| D-05 | 回滚 | P0 | 回滚到上一个成功版本。触发条件：Degraded > 5min。策略：手动触发（默认）/自动触发（可选，按服务级配置）。prod 自动回滚免审批；手动回滚到上一个 healthy 版本需 operator/admin 权限；回滚到指定历史版本需审批。平台通过 Argo CD API 执行回滚 |
| D-06 | 滚动发布 / 蓝绿 / 金丝雀 | P2 | 高级发布策略 |
| D-07 | 服务冻结/解冻 | P1 | 冻结后阻止该服务的新部署和配置变更，用于紧急止血和维护窗口。支持按维度冻结：全局（所有服务）、环境（某环境所有服务）、单服务+环境。进行中的部署不受影响 |
| D-08 | 服务接入流程 | P0 | 新服务接入平台的标准流程：注册 Service → 从模板目录选择部署模板（Helm Chart / Ansible Role OCI 制品）+ 版本 → 关联 Harbor 项目 → 初始化环境配置 → 创建 Argo CD Application |
| D-09 | 部署取消 | P1 | 审批前可取消部署请求；部署进行中不可取消（等待完成或失败） |
| D-10 | 部署前置校验 | P0 | 部署前校验镜像在 Harbor 中存在、Chart 版本可用，校验失败拒绝部署 |
| D-11 | 部署历史查询 | P1 | 查询某服务在某环境的部署历史记录（谁/什么时候/什么版本/结果） |
| D-12 | 平台级发布窗口 | P1 | admin 可设置全局发布冻结期（如节假日），冻结期内阻止 pre/prod 部署，operator 以上可申请紧急豁免 |
| D-13 | 版本漂移检测 | P1 | 平台每日扫描所有服务版本分布，检测 dev→prod 版本差距过大的异常服务，通过 Webhook + 站内消息通知 |

### 3.3A 模板管理

| ID | 功能项 | 优先级 | 描述 |
|----|--------|--------|------|
| T-01 | 构建模板管理 | P1 | 管理构建模板（Argo WorkflowTemplate），支持平台创建并发布为 OCI 制品，也支持注册外部 OCI 制品。模板版本不可变，支持归档 |
| T-02 | K8s 部署模板管理 | P0 | 管理 K8s 部署模板（Helm Chart），支持平台创建并发布为 OCI 制品，也支持注册外部 OCI 制品。服务接入时从模板目录选择绑定 |
| T-03 | 物理节点部署模板管理 | P1 | 管理 Ansible Role 模板，支持平台创建并发布为 OCI 制品，也支持注册外部 OCI 制品 |
| T-04 | 模板版本管理 | P0 | 每次发布生成不可变版本（semver），支持版本列表、归档旧版本、查看变更历史 |
| T-05 | 模板参数 Schema | P1 | 模板定义参数 schema（类型、默认值、约束），服务接入时按 schema 渲染配置表单 |

## 4. 非功能需求

| 维度 | 目标 |
|------|------|
| **规模** | 100+ 微服务，单 K8s 集群，持续增长 |
| **可用性** | 平台自身 SLA 99.5%（允许计划外停机 < 3.6h/月）；平台不可用时**不影响线上服务运行** |
| **性能** | 单次部署 < 5min；支持并行部署多个服务；API p95 < 200ms |
| **安全** | prod 环境操作需审批 + 审计；网络隔离；OIDC 认证 |
| **可扩展** | 新服务接入成本低（模板化）；平台水平扩展 |
| **可维护** | Go 单体，清晰模块边界 |

## 5. Non-Goals

| 不做 | 原因 |
|------|------|
| 自建 CI 引擎 | 构建由 Argo Workflows 执行，平台只触发和跟踪 |
| 自建监控系统 | 对接现有 Prometheus，不重复造轮子 |
| 工作流引擎 | 审批是单级的，不做复杂流程编排 |
| 日志聚合平台 | 对接现有 Loki/ES，平台只做跳转和查询入口 |
| 代码仓库 | Git 由现有 GitLab 管理 |
| 多租户隔离 | 当前阶段单组织运营，数据层已预留 org_id，暂不实现多组织管理界面 |
| 服务依赖拓扑 | 当前阶段不涉及，后续按需 |
| 多集群管理 | 当前阶段单集群，后续按需 |
| 对接 Vault | 当前阶段 KMS 信封加密已满足需求 |
| 镜像签名验证 | 后续按需引入 Cosign |

## 6. 验收标准

| 维度 | 标准 |
|------|------|
| 部署能力 | 能通过平台部署至少 3 个试点服务到 4 个环境，支持 Helm Chart |
| 物理机兼容 | 能通过平台部署 Ansible Role 到物理机 |
| 审批 | prod 环境部署必须经过审批，审批通知送达 < 1min |
| 权限 | 不同角色只能执行权限范围内的操作 |
| 审计 | 所有部署/配置变更操作可追溯 |
| 回滚 | 自动回滚触发 < 6min（Degraded 5min + 执行 1min）；手动回滚 < 2min |
| 状态可见 | 能实时看到部署进度和服务健康状态 |
| 集群管理 | 能纳管 K8s 集群并执行部署 |
| API 性能 | API p95 < 200ms |
| 审计完整性 | 100% 写操作有审计记录，hash chain 校验通过 |
